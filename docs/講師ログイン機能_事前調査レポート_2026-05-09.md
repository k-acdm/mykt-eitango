## 2. 既存認証ロジック確認

### サーバー側（[gas/Code.js:4867](../gas/Code.js)）
```javascript
function _verifyAdmin(password) {
  const stored = PropertiesService.getScriptProperties().getProperty('ADMIN_PASSWORD');
  return !!stored && password === stored;
}

function adminLogin(params) {
  if (!_verifyAdmin(params.password)) return { ok: false, message: 'パスワードが違います' };
  return { ok: true };
}
```
- **シングル パスワード方式**：Script Properties に `ADMIN_PASSWORD` 1 つだけ
- **ハッシュ化なし**：平文比較
- **セッション管理なし**：毎リクエストにパスワード送信
- **使用箇所**：全 admin API の冒頭で `_verifyAdmin(params.password)` チェック（既に統一済 = リファクタが楽）

### クライアント側（[admin.html:1423](../admin.html)）
```javascript
function doAdminLogin() {
  gasGet({ action: 'adminLogin', password: pw }).then(res => {
    if (res.ok) {
      sessionStorage.setItem('adminPassword', pw);  // ⚠️ 平文保存
      goAdminDashboard();
    }
  });
}
```
- **`sessionStorage.adminPassword`**：タブ閉で自動失効（OK）
- **平文保存**：sessionStorage は origin 単位なので比較的安全だが XSS 脆弱性あれば漏洩
- **認証エラー時の自動ログアウト**：`if (/認証|パスワード/.test(msg)) { showToast('セッションが切れました', 'error'); doAdminLogout(); }`（admin.html:1173 など多数）

### 既存 Teachers 基盤（CLAUDE.md #187、commit `2be9824`）
**既に実装済みだが現認証フローでは使われていない**：
- Teachers シート 6 列：`teacherId | teacherName | password | role | displayNickname | active`
- 初期データ：`T001 / 福地 / TEMP_PASSWORD_CHANGE_ME / 'admin' / 'ふく先生' / true`
- `sendTeacherMessage` の中で `_lookupTeacherDisplayNickname(senderId)` で表示名スナップショット取得に**だけ**使われている
- `password` 列は **未使用**（現認証は ADMIN_PASSWORD）
- `role` 列は `'admin'` だが、現状コード上で参照する場所はゼロ

→ Phase 1 で「Teachers の `password` / `role` を**実認証に使う**」拡張が必要。

## 3. 権限設計案 A〜C の比較

### 案 A：シンプルな 2 段階権限（オーナー / 講師）

**Teachers の `role` 列を 'owner' / 'teacher' の 2 値に**
```
T001 福地     [pw_hash] owner   ふく先生 true
T002 山田太郎 [pw_hash] teacher 山田先生 true
T003 佐藤花子 [pw_hash] teacher 佐藤先生 true
```

API 内のガード：
```javascript
function _requireOwner(password)   { return _verifyTeacherRole(password, 'owner'); }
function _requireTeacher(password) { return _verifyTeacherRole(password, ['owner','teacher']); }
```

| 評価軸 | 評価 |
|---|---|
| 実装コスト | 🟢 小（ガード 2 種、ロジックも単純） |
| 運用の明快さ | 🟢 高（誰でも理解可、ふくちさん vs 講師の二択） |
| 拡張性 | 🟡 中（後で細分化したくなった時に再設計） |
| UI 複雑度 | 🟢 低（オーナーは全表示、講師は隠す画面/ボタンを CSS/JS で制御） |
| **総合** | **シンプル・実用的・最初の選択肢として最有力** |

### 案 B：機能別の権限フラグ
```
Teachers シートに role ではなく機能別 boolean 列を追加：
canGrantHp / canEditTopics / canDeleteData / canViewRecording / canDownloadRecording / ...
```

| 評価軸 | 評価 |
|---|---|
| 実装コスト | 🔴 大（10+ 列、各 API でフラグごとガード） |
| 運用の明快さ | 🔴 低（「この講師は何ができるんだっけ」と毎回確認）|
| 拡張性 | 🟢 高 |
| UI 複雑度 | 🔴 高（権限編集 UI が複雑） |
| **総合** | **将来 100 講師規模なら最適だが、現段階では過剰設計** |

### 案 C：ロールベース（Role-based、4 階層）
```
'owner'         ふくちさん（全機能 + 講師管理）
'senior_teacher' 主任講師（HP 付与・問題入力可、講師管理不可）
'teacher'        一般講師(提出物確認 + コメント、HP 付与・問題入力不可)
'observer'       参観者・新人（読み取りのみ）
```

ロール毎に許可される機能セットを GAS 側で定義（ロール → 機能の matrix）：
```javascript
const ROLE_PERMISSIONS = {
  owner: { all: true },
  senior_teacher: { hpGrant: true, editTopics: true, viewRecording: true, downloadRecording: false, ... },
  teacher: { hpGrant: false, viewRecording: true, addComment: true, ... },
  observer: { readOnly: true }
};
```

| 評価軸 | 評価 |
|---|---|
| 実装コスト | 🟡 中（matrix 定義 + 1 ガード関数 + UI 分岐） |
| 運用の明快さ | 🟢 高（役職名で直感的に理解可）|
| 拡張性 | 🟢 高（新ロール追加は matrix に 1 行追加）|
| UI 複雑度 | 🟡 中（ロールごとの画面差分は明確）|
| **総合** | **最もスケーラブル、塾の規模が成長するなら最適** |

### クロ評価（公平な観点で）

ふくちさんの判断ポイントは **「現時点で講師は何人になりそうか」**：
- **2〜3 人で当面安定** → **案 A** が最良（実装コスト最小、ふくちさん vs 講師で十分）
- **将来 5 人以上、新人/ベテランの区別を付けたい** → **案 C**（owner / senior_teacher / teacher の 3 階層が現実的）
- **個人ごとに細かく権限を設定したい（特殊運用）** → 案 B

→ **案 A から始めて、必要なら案 C に発展させる**段階的アプローチが最も現実的。**`role` 列を `'owner' | 'teacher'` の 2 値で開始しておけば、後から `'senior_teacher'` を足すのも 1 行追加**で済む（既存データには影響なし）。

## 5. ログイン UI 設計の選択肢

### 案 X：teacherId + パスワード（個別認証、案 A 推奨）
```
┌─────────────────┐
│  講師ID  T001    │
│  パスワード ●●●●│
│  [ ログイン ]    │
└─────────────────┘
```
- pros: 各講師が個人パスワードを持つ、操作ログでも誰の行動か特定可
- cons: 入力が 2 段階

### 案 Y：講師選択（プルダウン）+ パスワード
```
┌─────────────────────┐
│ 名前 [▼ ふく先生]   │
│ パスワード ●●●●      │
│ [ ログイン ]         │
└─────────────────────┘
```
- pros: 講師ID を覚えなくて良い、UX 自然
- cons: **ログイン前に Teachers シートから講師リストを取得する必要**（無認証エンドポイントが必要）→ 講師の名前が外部に漏れる軽微なリスク。`active=true` のみ返す等で対処可

### 案 Z：従来通り単一パスワード（**非推奨**）
- 講師全員で同じパスワード共有、誰が何をしたか分からなくなる
- 操作ログの意味が消える

→ **案 X（teacherId + パスワード）が個人特定 + シンプルさのバランスで最良**。「講師ID は社員番号のような感覚で覚えられる」。Teachers シート編集時にふくちさんが各講師に番号を割り振る運用。

## 6. 操作ログの必要性評価

### 推奨：**実装すべき**（Phase 4、強く推奨）

#### 理由
- HP 手動付与のような「直接生徒データを変更する」操作は、後から「誰が・いつ・誰に・なぜ付与したか」が追跡できないと運用上の問題になる
- ふくちさんが「あれ、誰が付与したっけ？」と感じるシーンは確実に発生する
- 操作ログは **書き込み系 13 関数のみ** ガードに 1 行追加するだけで実装可（読み取り系は対象外）

#### 設計案
新シート `TeacherActions`（6 列）：
```
timestamp | teacherId | teacherNickname | actionType | targetSummary | details
```
例：
```
2026-05-08 21:30:15 | T002 | 山田先生 | HP_GRANT       | sid=22029 +500HP   | reason="授業内テスト合格"
2026-05-08 21:31:42 | T001 | ふく先生 | TOPIC_ADD      | wabun1 2026-05-12週| -
2026-05-08 21:35:08 | T002 | 山田先生 | COMMENT_ADD    | sango sid=24039     | "良い表現！"
```

GAS 側に共通関数 `_logTeacherAction(teacherId, actionType, targetSummary, details)` を追加し、書き込み系 API の冒頭で呼ぶ。

### スコープ
- 必須：HP_GRANT / MESSAGE_SEND_INDIVIDUAL / MESSAGE_SEND_ALL（高リスク）
- 推奨：TOPIC_ADD / COMMENT_ADD / TEACHER_LOGIN / TEACHER_LOGOUT（運用観察用）
- 不要：読み取り系（ノイズになる）

操作ログの閲覧 UI は owner 専用画面（Phase 3 と同時実装）。

## 7. Teachers シート初期化の方針

### 既存実装（CLAUDE.md #187、commit `2be9824`）
```javascript
// gas/Code.js:5123
function ensureTeachersSheet() {
  // 既存ならスキップ
  // T001 が無ければ追加：['T001', '福地', 'TEMP_PASSWORD_CHANGE_ME', 'admin', 'ふく先生', true]
  // ⚠️ T001 のパスワードを Teachers シート上で安全な値に書き換えてください（手動）
}
```

### 評価
- ✅ シート構造はそのまま使える（teacherId / teacherName / password / role / displayNickname / active の 6 列）
- ✅ 冪等
- ⚠️ `role: 'admin'` になっているが、案 A 採用なら `'owner'` に統一推奨（用語の整合）→ 既存 T001 行は手動修正、ensureTeachersSheet の new 行も `'owner'` に変更
- ⚠️ パスワードが**平文保存**：Phase 1 で**ハッシュ化**を導入推奨

### パスワード保存形式の推奨

| 方式 | pros | cons | 評価 |
|---|---|---|---|
| 平文 | 実装簡単 | シート漏洩 = 全パスワード漏洩 | 🔴 非推奨 |
| **SHA-256 + salt（GAS の Utilities.computeDigest）** | 漏洩しても元のパスワード不明、GAS 標準機能 | やや実装増 | 🟢 **推奨** |
| bcrypt | 業界標準・最強 | GAS で実装する必要あり、ライブラリなし、CPU 重い | 🟡 過剰 |

→ **SHA-256 + salt（生徒IDをsalt 兼用 or 固定 salt）**。`Utilities.computeDigest(Utilities.DigestAlgorithm.SHA_256, salt + password)` で十分。Phase 1 で実装。

### 初期パスワード設定フロー
1. ふくちさんが Teachers シートに新規行を追加（teacherId, teacherName, role, displayNickname, active のみ入力、password は空）
2. `setInitialTeacherPassword(teacherId, plaintext)` を GAS エディタから手動実行 → ハッシュ化して書き込む
3. 講師に teacherId と初期パスワードを伝達
4. 初回ログイン後、講師が自身でパスワード変更（Phase 3 で UI 提供）

## 8. 先生メッセージ機能との統合

### 現状（CLAUDE.md #187、commit `2be9824`）
```javascript
function sendTeacherMessage(params) {
  if (!_verifyAdmin(params.password)) return { ok: false, message: '認証エラー' };
  const senderId = String(params.senderId || 'T001');  // ← T001 がデフォルト
  // ...
}
```

ふくちさんの ADMIN_PASSWORD で認証 → senderId は params で指定（基本 T001）→ 表示名は Teachers シートからスナップショット。
→ **「ふくちさんが代理で T001 として送信」という現状運用には十分だが、講師ログイン後は不整合**。

### Phase 1 で必要な改修
- `sendTeacherMessage` の認証を `_verifyTeacher(params.teacherId, params.password)` に変更
- senderId は **params から渡さず、認証された teacherId を自動で使う**（なりすまし防止）
- フロント側 sessionStorage には `adminPassword` ではなく `teacherId` + `password` を保存

### 全体送信の制限（Phase 2 で実装）
- 'all' 送信は owner ロールのみ
- teacher ロールが all を要求したら 403 エラー

### 結論
**先生メッセージ機能は Phase 1〜2 と同時に「ロールベースに改修」で自然に着地**。CLAUDE.md #187 で「次回スレ最優先タスクとして長期保留中」と書かれているが、講師ログイン Phase 1〜2 をやれば**自動的にメッセージ機能の正式運用が開始**できる。

→ ふくちさんへ：「講師ログイン Phase 1〜2 = 先生メッセージ機能のリリース」という見方もできる。

## 9. 段階的実装の進め方提案（Phase 1〜6）

### Phase 1：認証基盤（**最重要**、~3〜4 時間）

**目的**：ADMIN_PASSWORD ベース → Teachers シートベースへ移行。ふくちさんと講師を区別できるようにする。

**実装範囲**：
- `_verifyTeacher(teacherId, password)` 新規追加（ハッシュ照合）
- `_getTeacherInfo(teacherId)` で role / displayNickname を取得
- `_passwordHash(plaintext, salt)` ヘルパー（SHA-256）
- `setInitialTeacherPassword(teacherId, plaintext)` GAS エディタ実行用
- `adminLogin` を `teacherLogin` にリネーム（または `adminLogin` 維持で内部処理だけ Teachers ベース化）
- 全 admin API の `_verifyAdmin(params.password)` を `_verifyTeacher(params.teacherId, params.password)` に置換（28 関数）
- フロント側 sessionStorage を `{teacherId, password, role, displayNickname}` の 4 値に拡張
- ログイン UI に「講師ID」入力フィールド追加
- ログアウト UI に「（〇〇先生としてログイン中）」表示

**互換性**：T001 のみのまま運用すれば、ふくちさんは引き続き「自分の teacherId + パスワード」で従来通り使える。新講師は ensureTeachersSheet 後にふくちさんが行追加するだけ。

**コミット粒度**：1 commit
**規模**：+200 / -50 行

### Phase 2：ロール別権限制御（~2〜3 時間）

**目的**：機能ごとに owner / teacher のアクセス制限を実装。

## 12. 重要な発見（補足）

### 既存設計の良い点
- `_verifyAdmin` が**全 admin API で統一して呼ばれている**（28 関数）→ Phase 1 のリファクタが楽
- `sessionStorage.adminPassword` の認証エラー時の自動ログアウトが**全 API 呼び出しで実装済み**（admin.html:1173 等多数）→ teacher ベースに切り替えても流用可
- **Teachers / TeacherMessages / MessageReads シートが既に存在**（commit `2be9824`、ふくちさん側で `ensureTeachersSheet()` 実行済の場合）→ Phase 1 で「使い始める」だけ

### 既存設計の懸念点
- パスワードが **平文保存**（gas/Code.js:5118）→ Phase 1 でハッシュ化必須
- `sendTeacherMessage` の `senderId = params.senderId || 'T001'` ← **クライアントが任意の senderId を指定できる**現状（gas/Code.js:5209）。ふくちさん 1 人運用なら問題ないが、講師ログイン後は**なりすまし送信できる脆弱性**。Phase 1 で「認証された teacherId を強制使用」に修正必須

### 講師数によるアプローチの違い
- **講師 2〜3 人なら**：Phase 1 + 2 のみで運用開始可（GAS エディタで Teachers シート手編集、Phase 3 講師管理 UI は後回し可）
- **講師 5 人以上なら**：Phase 3 講師管理 UI を Phase 1〜2 と同セッションで実装推奨（GAS エディタ編集が運用負荷になる）
