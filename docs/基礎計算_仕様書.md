## 1. 概要

### 1.1 目的・位置づけ

「基礎計算」は、マイ活アプリに追加する新コンテンツ。春日部アカデミーで長年使われてきた紙教材「計算級別トレーニング」（1級〜20級）をベースに、**事前生成された大量の問題DB**（Python + SymPyで機械的に生成・検証済み）から毎回ランダムに出題し、生徒が紙に答えを書いて写真送信、OCRで採点してHPを付与する仕組み。

**このコンテンツの特徴**

- **紙教材のデジタル化＋膨大な問題バリエーション**：各級数百〜千問を事前生成しているため、同じ生徒が同じ問題に再遭遇することはほぼない
- **計算の正確性を100%保証**：SymPy（Pythonの数式処理ライブラリ）で解答を機械検証済み。AIのような「もっともらしい誤答」は発生しない
- **待ち時間ゼロ**：DBから引くだけなので、問題表示は瞬時
- **答えは紙に書く**：途中計算・筆算の訓練を継続（タブレット上の入力では失われがちな基礎体力を維持）
- **管理画面で初回答案を確認可能**：字の丁寧さ・途中計算の有無を先生がチェックできる

**方式選定の経緯**

当初は Gemini API によるリアルタイム問題生成を検討したが、以下の理由で「Python + SymPy による事前生成方式（A案）」に変更した：

- LLMは数式を「もっともらしい形」で作れても、計算結果の正確性は保証できない
- 特に小数乗除の位取り、分数の約分、正負の数と累乗の複合、因数分解、平方根の有理化などでミスが発生しやすい
- 計算科目で「AIが間違った問題を出す」ことは、生徒の信頼を決定的に損なう
- SymPyは厳密計算ライブラリで、有理数・代数式・因数分解・平方根・二次方程式すべてを正確に扱える

### 1.2 対象学年

小学校高学年〜中学生

### 1.3 既存コンテンツとの違い

| コンテンツ | 出題形式 | 答え方 | 採点方式 |
|---|---|---|---|
| 英単語RUSH | 固定DB（英検5級〜準1級） | 画面入力 or 書き取り写真 | 完全一致 or Vision OCR |
| 三語短文 | お題日替わり | 画面入力（英文） | AI評価（Gemini） |
| 和文英訳① | 固定DB（4/27運用開始） | 画面入力（英文） | AI評価（Gemini） |
| **基礎計算** | **事前生成DB（Python+SymPy、各級数百〜千問）** | **紙に書く → 写真送信** | **DB保持の厳密解答とOCR照合** |

基礎計算は、**事前生成した大量問題DBから毎回ランダムに出題する**方式で、実質的に「毎回違う問題」を提供しつつ計算精度を100%保証する。

### 2.2 実施の流れ

1. 生徒が上記20種類の中から1つ選ぶ
2. 5題にするか10題にするかを画面上のボタンで選択。獲得HPもボタン上に表示
3. 内容に応じた問題をDBからランダムに抽出し、画面に表示（MathJax）
4. 生徒はそれを見て、問題番号と答えを紙に書き、解き終わったら写真に撮って送信
5. 「この内容で良いですか？」の確認表示
6. DBに保持されている厳密解答と照合し、⭕❌を返す
7. 正答率80%以上で合格 → HP獲得
8. 不合格の場合は、間違えた問題だけを再度画面に表示し、生徒がそれを解いて再送信
9. 再送信分が全問正解なら合格 → HP獲得
10. 合格するまで繰り返す
11. 合格時のHP獲得画面は、三語短文や和文英訳①と同様のものを流用

### 2.3 獲得可能HP

**素点HP**（連続実施ボーナス付与前のHP）

| セット | 合格時の素点HP |
|---|---:|
| 5題セット | 50HP |
| 10題セット | 100HP |

**1日の上限**

- 素点HPの合計が **1日100HPまで**
  - 10題×1セット = 100HP（上限到達）
  - 5題×2セット = 100HP（上限到達）
  - 上記以降に解いても素点HPは加算されない

**連続実施ボーナス倍率**

- 既存コンテンツと同様の倍率を適用
- 計算式：`獲得HP = 素点HP × Math.ceil(streak / 7)²`
  - 1〜7日連続：×1
  - 8〜14日連続：×4
  - 15〜21日連続：×9
  - 22〜28日連続：×16 …

**例**：連続10日目の生徒が10題セット合格 → `100 × Math.ceil(10/7)² = 100 × 4 = 400HP`

## 3. データ設計

### 3.1 スプレッドシート：新規シート構成

基礎計算コンテンツ用に、以下の4シートを新規作成する（問題DB本体＋運用シート3つ）。

#### ① `KisoQuestions`（問題マスタDB）

全20級 × 数百〜千問を事前生成して格納。稼働中は**読み取り専用**。

| 列名 | 型 | 説明 |
|---|---|---|
| questionId | string | 問題ID（`q_{rank}_{連番6桁}`、例：`q_14_000123`） |
| rank | number | 級（1〜20） |
| rankName | string | 級の名称（例：「分数：四則混合」） |
| difficultyBand | string | 難易度帯（`A`〜`H`、紙教材の8セット構造を継承） |
| problemLatex | string | 問題のLaTeX文字列（MathJax表示用） |
| answerCanonical | string | 標準解（最も自然な表記） |
| answerAllowed | string | 許容表記のJSON配列（OCR誤読吸収用） |
| generatedAt | datetime | 生成日時 |

→ 詳細は §6（Python問題生成スクリプト仕様）で規定。

#### ② `KisoSessions`（セッション管理）

| 列名 | 型 | 説明 |
|---|---|---|
| sessionId | string | セッションID（`kiso_{studentId}_{timestamp}_{random}`） |
| studentId | string | 生徒ID |
| rank | number | 級（1〜20、仕様書のNo.と同じ） |
| count | number | 問題数（5 or 10） |
| questionIds | string | 抽出された問題IDのJSON配列（例：`["q_14_000123","q_14_000456",...]`） |
| status | string | `in_progress` / `passed` / `failed_retry` |
| attempts | number | 提出回数（1〜） |
| startedAt | datetime | セッション開始日時 |
| completedAt | datetime | 合格した日時（未合格なら空） |
| hpEarned | number | 獲得HP（素点。ボーナス適用前） |

※ 実際の問題本体は `KisoQuestions` を `questionIds` で参照する。`KisoSessions` には問題文を複製しない。

#### ③ `KisoHPLog`（獲得HPログ）

**方針**：既存の `HPLog` シートに `source = "kiso_{rank}_{count}"` のような値で書き込んでまとめる（英単語RUSH、三語短文、和文英訳①と同じ扱い）。
→ 新規シート不要。既存ロジックを流用できる。

#### ④ `KisoPhotos`（写真メタ情報）

| 列名 | 型 | 説明 |
|---|---|---|
| sessionId | string | セッションID |
| studentId | string | 生徒ID |
| rank | number | 級 |
| count | number | 問題数 |
| driveFileId | string | Google DriveのファイルID |
| shareUrl | string | 管理画面から閲覧するための共有URL |
| submittedAt | datetime | 提出日時 |
| deleteAfter | date | 削除予定日（submittedAt + 15日） |

→ セッション初回提出時のみ1行追加。再挑戦時の写真は保存しない・行も追加しない。

### 3.2 問題データの形式

#### KisoQuestions シート上の1行（DB本体）

```
questionId       : q_14_000123
rank             : 14
rankName         : 分数：四則混合
difficultyBand   : D
problemLatex     : \frac{1}{2} + \frac{2}{3} \times \frac{3}{4}
answerCanonical  : 1
answerAllowed    : ["1","1/1","\\frac{1}{1}","1.0"]
generatedAt      : 2026-04-28T15:00:00+09:00
```

#### セッション開始時にフロントへ送るJSON

GASがセッションを開始したとき、フロント（index.html）には以下のJSONを返す。

```json
{
  "sessionId": "kiso_S001_20260425103000_abc123",
  "studentId": "S001",
  "rank": 14,
  "rankName": "分数：四則混合",
  "count": 10,
  "questions": [
    {
      "no": 1,
      "questionId": "q_14_000123",
      "problemLatex": "\\frac{1}{2} + \\frac{2}{3} \\times \\frac{3}{4}"
    },
    {
      "no": 2,
      "questionId": "q_14_000456",
      "problemLatex": "\\frac{5}{6} - \\frac{1}{3}"
    }
  ],
  "createdAt": "2026-04-25T10:30:00+09:00"
}
```

**ポイント**：
- フロントには **問題文のみ** 送る（`answerCanonical`、`answerAllowed` は送らない）
- 解答情報をフロントに流すと、ブラウザのdev toolsから見えてしまうため
- 採点はGAS側で行う（§7）

**LaTeX記法の共通ルール**（DB生成時・表示時の統一）

- 分数：`\frac{分子}{分母}`（MathJaxで縦2段表示）
- 乗算：`\times`（×記号）
- 除算：`\div`（÷記号。ただし分数形式で表現できる場合は分数形式を優先）
- 累乗：`x^{2}`、`(-3)^{2}`
- 平方根：`\sqrt{2}`、`\sqrt[3]{8}`
- かっこ：`( )`、`\{ \}`（中括弧は `\left\{ \right\}` で大きさ自動調整）
- 帯分数：`1\frac{1}{2}`（視認性のため MathJax なら問題ないが、DBでは `3/2` のような仮分数で持ち、表示時に帯分数化するか、固定で仮分数にするかは §6 で決める）

**許容表記（answerAllowed）の作り方**

OCR誤読・生徒の表記ゆれを吸収するため、各問題に複数の表記パターンを持つ。SymPyで標準化した後、以下のバリエーションを機械生成する：

- 分数スラッシュ：半角 `/` と全角 `／`
- 分数 vs 小数：`1/2` と `0.5` （両方とも正しい場合のみ）
- 約分：既約 `1/2` と非既約 `2/4` `4/8` …（非既約を認めるかは §7 で決める）
- 符号：`-3` と `−3`（ハイフンとマイナス記号）
- 空白：あり・なし両方
- 平方根の簡約：`2\sqrt{2}` と `\sqrt{8}`（両方受理、ただし問題の意図により厳格化する場合あり）

### 3.3 セッション管理の方法

1. **セッション開始**：
   - 生徒が級と問題数を選択
   - GAS が `KisoQuestions` シートから該当級の問題をランダムに `count` 問抽出
   - 抽出した `questionId` 配列を `KisoSessions.questionIds` に保存（`status = in_progress`）
   - フロントには問題文のみ返却（§3.2 の JSON 形式）
2. **問題表示**：フロント（index.html）が受け取ったJSONをMathJaxで表示
3. **写真提出（初回）**：
   - Vision API で OCR 読み取り
   - GAS が `KisoSessions.questionIds` から問題IDを取得 → `KisoQuestions` から `answerAllowed` を参照 → OCR結果と照合
   - 合格判定（正答率80%以上）
   - `KisoSessions.attempts = 1`
   - **写真を Google Drive に保存、`KisoPhotos` に1行追加**
4. **合格時**：`KisoSessions.status = passed`、`KisoSessions.completedAt` と `hpEarned` を更新、`HPLog` に1行追加
5. **不合格時**：`KisoSessions.status = failed_retry`、間違えた問題のみ再表示
6. **再挑戦（2回目以降の提出）**：
   - Vision OCR → 間違えた問題の `answerAllowed` と照合
   - `KisoSessions.attempts` をインクリメント
   - **写真は保存しない**（初回のみ保存）
   - 全問正解で合格 → 4 と同じ処理

**ランダム抽出の重複防止**（将来拡張候補）

- 同じ生徒が同じ級で、直近のセッションと同じ問題IDに当たらないようにする
- 実装案：生徒ごとに直近N回（例：5回）の `questionIds` を保持し、そこから除外して抽出
- 初回リリース時は単純ランダムで十分（各級1000問あれば自然に重複は稀）

### 3.4 写真データのストレージ

#### 保存先：Google Drive

ふくちさんのGoogleアカウント配下に専用フォルダを用意。

- ルートフォルダ：`マイ活_基礎計算_答案写真`
- 年月でサブフォルダを切る：`マイ活_基礎計算_答案写真/2026-04/`
- ファイル名規則：`{生徒ID}_{級}_{sessionId}.jpg`
  - 例：`S001_14_kiso_S001_20260425103000_abc123.jpg`

#### 保存タイミング

- セッション**初回**の提出写真のみ保存
- 再挑戦時の写真は Vision OCR に渡した後、保存せず破棄

#### 保存期間と自動削除

- 保存期間：**送信日から15日間**
- 削除方法：**GAS の Time-based Trigger** で日次実行
  - 毎日深夜 **3:00〜4:00の間** に起動（生徒のアクセスが最も少ない時間帯）
  - `KisoPhotos` シートを走査し、`deleteAfter <= 今日` の行について：
    1. Google Drive 上の該当ファイルを削除（`DriveApp.getFileById(driveFileId).setTrashed(true)`）
    2. `KisoPhotos` シートの該当行を削除
  - **確認メッセージなし・無条件削除**

#### 管理画面からの閲覧

- 管理画面（admin.html）に「基礎計算・初回答案」メニューを追加
- 生徒一覧 → 生徒を選択 → その生徒の保存中写真一覧（級・提出日時・削除予定日・サムネ）
- サムネ or ファイル名クリックで `shareUrl` を開く（Drive のプレビュー画面）
- ※詳細UIは §5 で扱う

#### 容量試算

- 想定画像サイズ：1枚あたり約300〜800KB（スマホ撮影、中品質JPEG）
- 生徒20人 × 1日最大2セット（5題×2 or 10題×1、後者は1セット） = 最大40枚/日
- 15日保存 = 最大600枚 ≒ 最大480MB
- Google Drive 無料枠（15GB）で十分収まる

## 6. Python 問題生成スクリプト仕様

### 6.1 方針と全体像

**目的**：20級 × 各級数百〜千問の問題を事前に機械生成し、`KisoQuestions` シートに格納する。生成時に SymPy で厳密計算を行うため、解答の正確性は100%保証される。

**実装方針**

- 実装言語：**Python 3.11+**
- 主要ライブラリ：
  - `sympy`：数式処理・厳密計算
  - `random`：乱数（`random.seed()` で再現性確保）
  - `gspread` + Google Sheets API：スプレッドシート書き込み
- 実行環境：ふくちさんの塾PC or 自宅PC のローカル（Colab 可）
- 実行頻度：**開発時に1〜数回のみ**。稼働後は原則再実行不要。問題を追加・差し替えしたい場合のみ個別級の再生成。

**総生成量の目安**

| 項目 | 数値 |
|---|---:|
| 1級あたりの生成問題数 | 約500〜1000問（級による） |
| 20級合計 | 約15,000〜20,000問 |
| SymPyでの1問あたり生成時間 | 0.05〜0.5秒 |
| 総生成時間の見込み | **1〜3時間で完了** |
| スプレッドシート書き込み | 数分（バッチ書き込み） |

### 6.3 各級スクリプトの共通インターフェース

すべての `rank_XX_*.py` は以下の関数を持つ：

```python
def generate_problem(band: str, rng: random.Random) -> dict:
    """
    1問を生成する。

    Args:
        band: 難易度帯 ('A'〜'H'、紙教材の8セット構造に準拠)
        rng: 乱数生成器（再現性確保のため外部から注入）

    Returns:
        {
            "problemLatex": str,    # 問題のLaTeX文字列
            "answerCanonical": str, # 標準解
            "answerAllowed": List[str]  # 許容表記のリスト
        }
    """
```

### 6.4 共通モジュールの仕様

#### 6.4.1 `latex_utils.py`

SymPy 表現を LaTeX に変換する際の**表記統一**を司る：

- 分数：`\frac{a}{b}` 形式（SymPy標準の `\dfrac` ではなく `\frac` に統一）
- 乗算：`\times`（× 記号）
- 除算：基本は分数形式、ただし問題に `\div` が必要な場合は明示
- 累乗：`x^{2}`、`(-3)^{2}`（かっこ付き符号）
- 平方根：`\sqrt{8}`、`\sqrt[3]{27}`
- 帯分数：**DBの `answerCanonical` には仮分数で格納**（`3/2`）、ただし帯分数表記 `1\frac{1}{2}` も `answerAllowed` に含める（§6.8 決定1）

#### 6.4.2 `sympy_helpers.py`

各級で共通利用する SymPy ヘルパー：

- `to_rational(value)`：浮動小数 → 厳密分数（Rational）変換
- `is_integer_ratio(frac)`：整数か判定
- `reduce_fraction(num, den)`：既約分数化
- `sqrt_simplify(expr)`：平方根の簡約（`\sqrt{8}` → `2\sqrt{2}`）
- `expand_and_simplify(expr)`：展開と整理

#### 6.4.3 `answer_variants.py`

標準解から許容表記のバリエーションを機械生成：

- **分数スラッシュ**：`1/2` ⇔ `1／2`
- **空白**：`2x + 3` ⇔ `2x+3`
- **符号記号**：`-3` ⇔ `−3`（ハイフンとマイナス）
- **分数↔小数**：`1/2` ⇔ `0.5`（**有限小数になる場合のみ**。無限小数は分数のみ）
- **帯分数↔仮分数**：`3/2` ⇔ `1 1/2`（§6.8 決定1に基づき両方とも許容）
- **非既約分数**：`1/2` に対する `2/4`、`4/8` 等は**許容しない**（§6.8 決定2）
- **非簡約の平方根**：`2\sqrt{2}` に対する `\sqrt{8}` は**許容しない**（§6.8 決定3）
- **非有理化の平方根**：`\sqrt{2}/2` に対する `1/\sqrt{2}` は**許容しない**（§6.8 決定3）

**許容表記の生成ルールは採点仕様（§7）と密接に関連**するため、両者の整合を保つ。

#### 6.4.4 `band_config.py`

各級の A〜H バンドごとの生成問題数と難易度パラメータを一元管理：

```python
BAND_PLAN = {
    20: {
        "A": {"count": 100, "digits": 1, "terms": 2, "ops": ["+","-"]},
        "B": {"count": 100, "digits": 1, "terms": 2, "ops": ["×","÷"]},
        # ... H まで
    },
    19: { ... },
    # ...
}
```

### 6.5 各級の問題生成仕様（概要）

**詳細な紙教材パターン**は、Claude Code 実装時に級別トレーニング教材の画像（A〜H 全8セット分）を参照資料として渡す。ここでは各級の**構造の要点**だけ記載する。

| 級 | 名称 | 生成の主要ポイント |
|---:|---|---|
| 20 | 整数四則混合 | 1〜3桁整数、2〜3項、四則混合、括弧あり。A〜Hで桁数・項数・括弧の複雑さを段階的に上げる |
| 19 | 小数加減 | 1〜3桁の小数、2〜3項。D級以降は桁違い（`10.52 + 5.48`、`2 − 0.456`）。H級は3桁同士の3項 |
| 18 | 小数乗除 | A〜D は片方整数、E以降は小数×小数。位取り厳密、割り切れない割り算（四捨五入ルールなし、厳密値のみ） |
| 17 | 小数四則混合 | 括弧・3項混合必須。H級は `124 ÷ 0.04 + 0.6 × 90` のような高難度 |
| 16 | 分数加減 | A：同分母、B〜：異分母、D〜：帯分数・3項、F〜：**小数と分数の混在**（`7/15 + 0.9`） |
| 15 | 分数乗除 | 途中約分が必須。D〜：帯分数、F〜：小数との混在、G〜H：3項乗除 |
| 14 | 分数四則混合 | 括弧・四則混合、帯分数、小数混在、3項以上（前スレ確認済み） |
| 13 | 正負の数 加減 | A：括弧付き `(+9)+(+5)`、D：括弧なし `-47+58`、E〜：小数・分数との混在、G〜H：3項以上 |
| 12 | 正負の数 乗除 | B〜：累乗 `(-3)²`、`-3²×(-2)`、D〜：3項乗除、G〜H：累乗と分数の混在 |
| 11 | 正負の数 四則混合 | **最難関級**。累乗・括弧・中括弧 `{…}`、小数・分数混在。H級は高校入試レベル |
| 10 | 単位・比・割合 | **10問テンプレート構造**。①〜⑤単位換算（長さ/面積/容積/重さ）、⑥時間、⑦速さ、⑧比、⑨割合（小数・%・割の相互変換）、⑩割合応用。各スロットごとに複数パターンを用意してランダム抽出 |
| 9 | 式の計算・中1 | 文字式の同類項整理、分配法則、単項式の乗除 |
| 8 | 一次方程式・比例式 | `3x=21` のシンプルなものから `0.01(x+2) = 0.03 - 0.02x` のような小数係数、分数係数まで。比例式も含む |
| 7 | 式の計算・中2 | 多項式の加減・乗除、単項式の累乗、分数係数、筆算形式（縦書き加減）も含む |
| 6 | 連立方程式 | 加減法・代入法の両方。解は整数・分数両対応。SymPy の `solve` で検証 |
| 5 | 式の計算・中3 | 多項式の展開、分配法則の拡張 |
| 4 | 乗法公式 | `(x+a)(x+b)`、`(x+a)²`、`(x+a)(x-a)` の3パターン中心 |
| 3 | 因数分解 | 共通因数、`x² + bx + c`、`ax² + bx + c`（たすき掛け）、`x² - a²`、`x² ± 2ax + a²` |
| 2 | 平方根 | 簡約、加減、乗除、有理化、二重根号（必要なら）。SymPyで厳密計算 |
| 1 | 二次方程式 | 因数分解・平方完成・解の公式、すべてのパターン。整数解・無理数解・重解すべて対応 |

### 6.6 生成→DB投入のパイプライン

```
main.py 実行
 ↓
各級スクリプトをループ呼び出し
 ↓
generate_problem(band) を BAND_PLAN 回数分実行
 ↓
各問題について以下を実施：
  1. SymPyで問題式を生成
  2. SymPyで解答を厳密計算
  3. 生成式を SymPy で再計算し、解答と一致するかセルフチェック
  4. LaTeX に変換
  5. 許容表記を生成
 ↓
全問題を pandas.DataFrame に集約
 ↓
gspread で KisoQuestions シートに一括書き込み
 ↓
書き込み完了後、ログ出力（各級の生成数、失敗数、総時間）
```

### 6.7 検証と品質保証

生成スクリプト自体の正しさを保証するため、以下を自動実行する：

**① セルフチェック（生成時）**
各問題を生成した直後、以下を確認：
- `answerCanonical` を SymPy で再計算し、元の答えと一致するか
- `answerAllowed` 内のすべての表記が、SymPy的に同値か
- `problemLatex` がMathJaxで解釈可能な形式か（簡易構文チェック）

**② ランダムサンプリング目視（生成後）**
- 各級から10問ずつランダム抽出 → 人間（ふくちさん or Claude）が目視確認
- 合計 200問を目視 → 許容範囲なら本番投入

**③ 許容表記のカバレッジテスト**
- 生徒が書くであろう表記（既約・非既約、スラッシュ全半角、空白あり・なし）を想定テストケースとして用意
- 各級の代表問題10問で、テストケースすべてが合格扱いになるか自動確認

### 6.8 表記・採点ルールの決定事項

ふくちさんとの協議により、以下のルールを確定する。§7 採点ロジックもこれに従う。

#### 決定1：帯分数と仮分数の両方を許容

- 解答は **仮分数・帯分数の両方** を正解として認める
- 例：`3/2` も `1 1/2` も正解
- 理由：問題自体に両表記が混在する上、小学生も利用する想定のため
- DB上の `answerCanonical` は**仮分数**で格納（表記の一意性）。`answerAllowed` に帯分数表記も含める

#### 決定2：**既約分数のみ正解**（非既約は不正解）

- 生徒が `2/4` と書いた場合、`1/2` の正解としては**認めない**
- 理由：テストでは必ず ×。基礎計算の厳密さを保つ
- 実装：SymPy で `Rational(num, den)` に正規化し、分子・分母が GCD=1 である形のみ受理
- 例外：整数（`6/3 = 2` のように整数になる場合は整数表記 `2` を canonical とし、`6/3` は×）

#### 決定3：**無理数は簡約形のみ正解**（非簡約は不正解）

- 生徒が `\sqrt{8}` と書いた場合、`2\sqrt{2}` の正解としては**認めない**
- 理由：テストでは必ず ×
- 実装：SymPy の `sqrtdenest` や `radsimp` で最簡形を算出し、その表記のみ受理
- 有理化も必須：`1/\sqrt{2}` ではなく `\sqrt{2}/2` のみ正解

#### 表記ルール一覧（採点時の正解条件）

| 数値種別 | 正解条件 | 例 |
|---|---|---|
| 整数 | 整数表記 | `6`（`12/2` や `6.0` は×） |
| 分数 | 既約分数（仮分数・帯分数両方可） | `3/2` or `1 1/2`（`2/4` は×） |
| 小数 | 問題文脈による有限小数 | `0.5`、`3.14` |
| 平方根 | 簡約＆有理化済み | `2\sqrt{2}`（`\sqrt{8}` は×） |
| 比 | 最簡整数比 | `2:3`（`4:6` は×） |
| パーセント | 問題文で指定された形式 | `50%` or `0.5` |
| 負の数 | 符号付き | `-5`、`-3/4` |

## 7. 採点ロジック

### 7.1 全体フロー

```
生徒が写真送信
 ↓
GAS：画像を Vision API（DOCUMENT_TEXT_DETECTION）に投げる
 ↓
OCR結果：テキスト＋位置情報
 ↓
問題番号と答えのペアに分解（①＝○○、②＝○○…）
 ↓
各問題について KisoQuestions.answerAllowed と照合
 ↓
⭕❌を配列で返す
 ↓
正答率80%以上？
 ├─ Yes → 合格 → HP付与（§8）
 └─ No  → 不合格 → 間違えた問題のみ再挑戦
```

### 7.2 OCR（Vision API）

#### 利用API

既存の `VISION_API_KEY`（GAS側）を流用。英単語RUSH書き取り採点で使用中のものと同じ。

#### 呼び出し形式

- エンドポイント：`https://vision.googleapis.com/v1/images:annotate`
- タイプ：`DOCUMENT_TEXT_DETECTION`（手書き文字の認識精度が高い）
- 入力：生徒のアップロード画像（base64エンコード、事前にクライアント側でリサイズ済み）
- 出力：`fullTextAnnotation` に「ページ→ブロック→段落→単語→シンボル」の階層構造でテキストと座標

#### 位置情報の活用

紙に書かれた答えは以下のような形式を想定：
```
① 3/2
② 0.58
③ 2√2
...
```

Vision APIが返すテキストを「問題番号（①〜⑩）」で分割し、各番号に対応する答えを抽出する。

**分割ロジック**：
1. 全文テキストから `①`〜`⑩` を検出
2. 各番号間のテキストを当該問題の答えとして抽出
3. 答え部分から空白・改行を正規化

### 7.3 許容表記との照合ロジック

#### 基本方針

生徒の答え（OCR結果）を正規化 → `KisoQuestions.answerAllowed` 内のいずれかと完全一致するかを判定。

#### 正規化処理

照合前に、OCR結果と許容表記の両方に以下の正規化を施す：

| 正規化項目 | 処理 |
|---|---|
| 全角数字 → 半角 | `０` → `0`、`１` → `1` … |
| 全角スラッシュ → 半角 | `／` → `/` |
| 全角空白 → 半角 | `　` → ` ` |
| 空白の連続圧縮 | `1  1/2` → `1 1/2` |
| 前後の空白除去 | trim |
| 全角マイナス → 半角 | `−`、`ー`、`─` → `-` |
| 小数点 | `．` → `.` |
| パーセント | `％` → `%` |
| 平方根記号 | `√` → `\sqrt`、OCRで `V` と誤読される場合は文脈判定 |
| 分数表記 | OCRで `─` や `/` の判定揺れあり → 縦書き分数をパターンマッチで `a/b` 化 |

#### 照合の段階

1. **完全一致チェック**：正規化後の生徒答えが `answerAllowed` のいずれかと一致 → ⭕
2. **数値同値チェック（フォールバック）**：完全一致しない場合、生徒答えをSymPyでパース → 厳密解答と同値か判定
   - ただし§6.8の決定2・3に従い、**非既約分数・非簡約平方根は不正解**
   - この段階では「生徒答えを SymPy でパースして、Canonical化 → 厳密解答の Canonical と文字列一致」を確認
   - 例：生徒が `2/4` → Canonical化すると `1/2` → 正解 `1/2` と一致するが、**この経路は使わない**（§6.8決定2により `2/4` は×）
3. **OCR誤読の救済**：OCR信頼度が低いシンボル（例：`1` と `l`、`0` と `O`、`/` と `7`）は、許容表記側で両方を用意しておく

#### 重要：既約性・簡約性の厳格チェック

§6.8 の決定2・3 に従い、**表記そのものが既約・簡約形であるか**を判定する：

- 分数：正規表現で `a/b` を抽出 → `gcd(a, b) = 1` かチェック。`gcd > 1` なら ×
- 帯分数：`c a/b` を抽出 → `a < b` かつ `gcd(a, b) = 1` かチェック
- 平方根：`\sqrt{n}` の `n` が平方因子を含まないか、`a\sqrt{b}` の `b` が平方因子を含まないかチェック
- 有理化：分母に `\sqrt` を含む表記は ×

この厳格チェックは、**OCR後の正規化済み文字列に対して直接適用**する。SymPy によるパース結果ではなく、「生徒の書いた文字列そのもの」の形式で判定することが重要。

### 7.4 合格判定（80%ルール）

- 正答数 ÷ 総問題数 ≥ 0.80 → 合格
- 5題の場合：4問以上正解で合格（80%）
- 10題の場合：8問以上正解で合格（80%）

**境界ケース**：
- 5題中4問正解（80.0%）→ 合格
- 10題中8問正解（80.0%）→ 合格

### 7.5 再挑戦時の採点

#### 再挑戦フロー

1. 初回で不合格 → 間違えた問題のみを再表示（§4.8）
2. 生徒が再度写真を送信
3. **間違えた問題のみ** OCR → 照合
4. **全問正解で合格**（80%ルールは適用せず、間違えた問題は全て⭕でないと合格にならない）
5. 1問でも誤答 → さらに再挑戦

#### 実装上のポイント

- `KisoSessions.questionIds` は初回抽出時の全問題IDを保持
- 再挑戦時は「前回不正解だった問題IDのサブセット」を別途セッションステートに保持
- 再挑戦のたびに `KisoSessions.attempts` をインクリメント

### 7.6 エッジケースとフォールバック

#### ケース1：OCRが空欄を検出できない

- 生徒が①〜⑤のうち③を書き忘れ → OCR結果に③が欠落
- 処理：該当問題を「×（無回答扱い）」として採点

#### ケース2：OCRが問題番号を誤検出

- `①` を `(1)` と認識、`⑤` を `5` と認識、など
- 対策：許容範囲の番号表記（`①`、`(1)`、`1.`、`1、`、`1 `）を正規表現で吸収

#### ケース3：複数行に跨る答え

- `2\sqrt{2}` を `2` と `√2` で2行に分けて書く生徒
- 対策：OCRの段落情報を活用し、次の問題番号が出るまでを1つの答えとして結合

#### ケース4：OCR信頼度が全体的に低い

- 画像が暗い、ピンボケ、など
- 対策：Vision APIの応答に含まれる confidence score を監視。閾値以下の場合「写真が読めなかったみたい！もう一回撮ってみてね」のメッセージで再撮影を促す
- 閾値（暫定）：全体平均 confidence < 0.6 → 再撮影要求

#### ケース5：明らかに範囲外の数値（誤答だが惜しい場合）

- 例：正解が `1/2` で生徒が `1/3` と書いた → 完全に×
- 特別扱いしない（中間点なし）

### 7.7 採点に関する決定事項

| 項目 | 決定 |
|---|---|
| 空欄の扱い | 不正解（×）として扱う |
| OCR信頼度の低い画像 | 平均 confidence < 0.6 で再撮影要求。閾値は実機テストで最終調整 |
| 部分点 | なし。1問1⭕❌で統一 |

## 8. HP加算ロジック

### 8.1 素点HPの決定

合格時の素点HPは、セッションの問題数で決まる：

| 問題数 | 素点HP |
|---:|---:|
| 5 | 50 |
| 10 | 100 |

### 8.2 連続実施ボーナスの適用

マイ活アプリ既存コンテンツと同じ計算式を適用：

```
獲得HP = 素点HP × Math.ceil(streak / 7) ²
```

| 連続日数（streak） | 倍率 |
|---:|---:|
| 1〜7 | 1 |
| 8〜14 | 4 |
| 15〜21 | 9 |
| 22〜28 | 16 |
| … | … |

### 8.3 1日の素点上限：100HP

#### 上限の考え方

- **「素点HP」ベースで1日100HPまで**が上限
- 倍率適用後のHP（例：連続10日目で400HP）には上限をかけない
- 既存の他コンテンツ（英単語RUSH、三語短文、和文英訳①）から独立してカウント

#### 実装方式

GAS側で当日の累積素点HPを計算：

```javascript
// 擬似コード
function calculateTodaysKisoRawHP(studentId) {
  const today = getJSTDateString(); // "2026-04-25" 形式
  const logs = getHPLogByStudent(studentId);
  return logs
    .filter(log => log.source.startsWith("kiso_") && !log.source.endsWith("_practice"))
    .filter(log => log.dateJST === today)
    .reduce((sum, log) => sum + log.rawHP, 0);
}
```

**`rawHP` カラムの追加が必要**：

既存 `HPLog` シートには `hp`（倍率適用後）しか無い想定。基礎計算の上限判定のため、`rawHP`（素点）も併せて記録するカラムを追加する。

### 8.4 HPLog への記録フォーマット

#### 通常合格時（素点が上限内）

| カラム | 値 |
|---|---|
| timestamp | 2026-04-25T14:35:22+09:00 |
| studentId | S001 |
| source | `kiso_14_10`（rank 14 の 10題セット） |
| rawHP | 100 |
| hp | 400（= 100 × 4、連続10日目想定） |
| sessionId | kiso_S001_20260425103000_abc123 |

#### 上限到達後の練習モード合格時

| カラム | 値 |
|---|---|
| timestamp | 2026-04-25T15:50:10+09:00 |
| studentId | S001 |
| source | `kiso_14_5_practice` |
| rawHP | 0 |
| hp | 0 |
| sessionId | kiso_S001_20260425155000_xyz789 |

**このレコード**：
- 学習ログとして意味を持つ（管理画面で「練習実施あり」と見える）
- HP集計には加算しない（`hp = 0`）
- 連続日数のカウントには含める（学習実施日として）

### 8.5 境界ケース

#### ケース1：5題を3セット実施

- 1セット目：素点50HP → 通常記録
- 2セット目：素点50HP（累計100HP、ちょうど上限） → 通常記録
- 3セット目以降：**練習モード**（素点0HP、`_practice` サフィックス）

#### ケース2：5題合格後に10題挑戦

- 1セット目（5題）：素点50HP → 通常記録
- 2セット目（10題）：**素点は残り50HPまでのみ付与**（1日の素点HP上限100HPを厳守）
  - 実装：`獲得素点HP = Math.min(100, 100 - 既に獲得した素点)` = `min(100, 50)` = `50`
  - `rawHP = 50`、`hp = 50 × 倍率`
- 以降：練習モード

### 8.6 連続日数（streak）の計算

既存コンテンツと共通ロジックを使用：
- 生徒が当日初めて何らかのコンテンツで**HP獲得できるアクションを完了**した時点で、連続日数を判定・更新
- 練習モード（`hp = 0`）でも連続日数のカウントには含める

**「streak」の定義**（既存仕様を踏襲）：
- 日本時間の0:00を区切りとする
- 前日に1つ以上の学習実施があれば連続継続
- 前日に学習実施がなければ streak = 1 にリセット

### 8.7 HP加算に関する決定事項

| 項目 | 決定 |
|---|---|
| 5題×3セットの扱い | 1・2セット目はHP付与（累計100HPで上限到達）、3セット目以降は練習モード |
| 5題→10題の流れで素点超過時 | 10題セット側で残り素点（50HP）のみ付与、超過分は切り捨て。1日の素点HP上限100HPを厳守 |
| `HPLog.rawHP` カラム追加 | 追加する。既存レコード（他コンテンツ）は `rawHP = hp` で埋め戻し。将来の他コンテンツでの素点上限機能導入に備えたデータ構造統一のため |

## 10. 残課題・今後の拡張

### 10.1 確認事項の状況

すべての設計判断は確定済み：

- §6.8 表記・採点ルール（帯分数/仮分数両方許容、既約のみ、簡約形のみ）→ 決定済み
- §7.7 採点の詳細（空欄=×、OCR信頼度閾値=0.6、部分点なし）→ 決定済み
- §8.7 HP加算の詳細（5×3セット、5→10の超過、rawHPカラム）→ 決定済み

### 10.2 本実装で対応しないもの（意図的に見送り）

- 管理画面の写真ダウンロード機能（Drive で代替可）
- 管理画面での写真コメント機能（生徒フィードバック用）
- 連立方程式の途中式採点（答え（x, y）のみ採点）
- 問題の難易度自動調整（生徒の正答率に応じて出題バンドを動的変更）

### 10.3 将来拡張候補

#### 短期（リリース後1〜3ヶ月）

- **出題履歴の重複回避**：生徒ごとに直近N回の `questionIds` を保持して除外
- **級ごとの達成バッジ**：各級100問合格ごとに特別バッジ、マイカツ君が祝福

#### 中期（半年〜1年）

- **統計ダッシュボード**：管理画面で生徒別の級別進捗、苦手分野可視化
- **苦手問題の優先出題**：生徒が過去に間違えた問題パターンを重点的に出題
- **紙の宿題連動機能（メモリ#4）**：先生が紙宿題を写真登録 → 生徒が答案写真送信 → AI採点＋HP付与

#### 長期（1年以降）

- **アバター機能との連携（メモリ#3）**：基礎計算合格でHP獲得 → アバター着せ替えに使える
- **学年区分ランキング（メモリ#1）**：基礎計算は小〜中対応なので、学年別ランキングに重要な寄与

### 10.4 運用上の注意

#### 問題DBのメンテナンス

- 生成した問題に誤りが発見された場合：
  1. `KisoQuestions` シートの該当行を「削除」列フラグで無効化
  2. `startKisoSession` では無効化されていない行からのみ抽出
  3. 年1回程度、無効化された問題を物理削除

#### Gemini API への方針変更の可能性

- 本仕様ではリアルタイム生成を採用しないが、将来的にLLMの計算精度が向上した場合に再検討
- 再検討ポイント：
  - SymPy検証との併用ハイブリッド（生成はGemini、検証はSymPy）
  - 生徒の個別レベル診断に基づく問題生成（これはDB方式では困難）

#### 写真データの扱い

- 15日間の保存期間は、万が一のトラブル時の証跡としての意味もある
- 生徒のプライバシー保護のため、写真はGoogle Drive上でも「リンクを知っている人のみ」の共有設定とし、パブリック公開はしない
