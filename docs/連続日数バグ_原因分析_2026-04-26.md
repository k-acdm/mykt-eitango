# 連続日数バグ 原因分析レポート（2026-04-26）

## 事象サマリ

4/26 朝、保護者から「29 日連続だったのが 1 日に戻っている」との問い合わせ。複数の生徒（10 名以上）で `Students.STREAK` が異常な値にリセットされていることが判明。

### 観測パターン

- **完全リセット（STREAK=1）**：22025 笹川夏鈴 / 24018 早川康佑 / 24009 岩倉寿人 / 24027 片山史也（いずれも毎日合格中）
- **部分破損**：23030 髙山遥太 → STREAK=6、24030 松本凌玖 → STREAK=10
- **無傷**：22029 中綾音（35）、24003 古内伶奈（35）、23009 太田杏乃実（34）など

→ 全体障害ではなく **条件依存のバグ**。

## 根本原因

### 1. 主因：`_toDateStr` 正規表現の anchor 不足

[gas/Code.js:43-52](../gas/Code.js#L43)：

```javascript
function _toDateStr(val) {
  if (!val) return '';
  try {
    var s = String(val);
    if (s.match(/^\d{4}-\d{2}-\d{2}/)) return s.slice(0, 10);  // BUG: 末尾 $ なし
    return Utilities.formatDate(new Date(val), 'Asia/Tokyo', 'yyyy-MM-dd');
  } catch(e) { return String(val).slice(0, 10); }
}
```

正規表現 `/^\d{4}-\d{2}-\d{2}/` に末尾アンカー `$` がないため、ISO 8601 形式（例：`"2026-04-25T15:00:00.000Z"`）にもマッチし、先頭 10 文字 `"2026-04-25"` を切り出してしまう。

### 2. 引き金：Day 2 Stage 2（4/24 commit `e6d8398`）の Students キャッシュ

`cache_students_values` キーで Students シート全体をキャッシュ。

- 書き込み：`JSON.stringify(values)`
- 読み出し：`JSON.parse(hit)`

`Date` オブジェクトは JSON シリアライズで **UTC ISO 文字列** に変換される。

```text
LAST_LOGIN セル（Date 型）= JST 2026-04-25 0:00
  ↓ getValues
Date object (UTC 2026-04-24 15:00:00)
  ↓ JSON.stringify
"2026-04-24T15:00:00.000Z"
  ↓ JSON.parse + _toDateStr (バグあり)
"2026-04-24"   ← 本当は "2026-04-25"（JST）。1 日ズレる
```

### 3. 二次要因：`saveAttempt` の setValues 範囲が STREAK 列を含む

[gas/Code.js:619-622](../gas/Code.js#L619)：

```javascript
const preservedStreak = sRows[i][COL_STREAK];  // ← cache 経由（stale 可能性）
sSheet.getRange(i + 1, COL_CLEARED + 1, 1, COL_LAST_TEST - COL_CLEARED + 1)
      .setValues([[newCleared, now, newHP, preservedStreak, today]]);
//                                          ^^^^^^^^^^^^^^ STREAK 列に書き戻し
```

cache が stale だと、`saveAttempt` が **正しい STREAK 値を上書きしてしまう**。

## バグ発動の連鎖シナリオ（22025 笹川の例）

### 4/24（Day 2 Stage 2 デプロイ後）

1. cache miss → loader で Students シートを読む
2. `LAST_LOGIN` セルが Date 型のため Date オブジェクトとして取得
3. `JSON.stringify` で `"2026-04-22T15:00:00.000Z"` に変換、cache に保存
4. cache hit 時、`_toDateStr` が `"2026-04-22"` を返す（本当は JST `"2026-04-23"`）
5. `today = "2026-04-24"`、`missedDays = 2` と誤判定 → `streak = 1`（リセット枝）
6. シートに `STREAK=1`、`LAST_LOGIN="2026-04-24"`（文字列）を書き込み
7. 続く `saveAttempt` が `preservedStreak = 1` を STREAK 列に書き戻し → 永続化

### 4/25 以降

- cache miss が起きるたびに同じ off-by-one が再発
- セル書式が Date のため、設定した `"2026-04-24"` 文字列も Sheets が再度 Date に自動変換
- `_toDateStr` のバグが残り続ける限り、毎日 STREAK=1 にリセットされる

## なぜ生徒によって挙動が違うか

`LAST_LOGIN` セルの **書式が生徒ごとにばらついている**ため：

- **被害生徒**：セル書式が「日付」自動フォーマットされている → JSON roundtrip で off-by-one
- **無傷生徒**：セル書式が「テキスト」または「自動」のまま文字列保持 → JSON roundtrip しても "2026-04-25" のまま
- **部分破損**：cache miss/hit の偶発的タイミングで一部の日のみバグ発動

## 修正方針

### 主因対策（必須）

`_toDateStr` の正規表現に末尾アンカーを追加し、ISO datetime は `Utilities.formatDate` 経由で JST に再フォーマット：

```javascript
function _toDateStr(val) {
  if (!val) return '';
  try {
    var s = String(val);
    // 純粋な日付文字列（_todayJST() が返すフォーマット）
    if (s.match(/^\d{4}-\d{2}-\d{2}$/)) return s;
    // _nowJST() フォーマット（タイムゾーンなしの日時文字列）
    if (s.match(/^\d{4}-\d{2}-\d{2}\s/)) return s.slice(0, 10);
    // Date オブジェクトまたは ISO datetime（タイムゾーン付き） → JST で再フォーマット
    return Utilities.formatDate(new Date(val), 'Asia/Tokyo', 'yyyy-MM-dd');
  } catch(e) { return String(val).slice(0, 10); }
}
```

### Defense in depth（推奨）

- **saveAttempt** の setValues 範囲から STREAK 列を除外
  - 旧: D-H 5 列（CLEARED, UPDATED, HP, **STREAK**, LAST_TEST）
  - 新: D-F 3 列（CLEARED, UPDATED, HP）+ H 単発（LAST_TEST）
- **loginStudent** の setValues 範囲から LAST_TEST 列を除外（同じ stale 書き戻しリスクの排除）
  - 旧: E-I 5 列（UPDATED, HP, STREAK, **LAST_TEST**, LAST_LOGIN）
  - 新: E-G 3 列（UPDATED, HP, STREAK）+ I 単発（LAST_LOGIN）

### 他の Day 2 改善のリスク評価

| 改善箇所 | 評価 |
|---|---|
| `_getPrevDayCount` 末尾 200 行読み | OK（Attempts 直読み、cache 不経由） |
| `_getQuestionRowsForLevel` Questions キャッシュ | OK（日付処理に無関係） |
| `_getStudentsValues` キャッシュ自体 | OK（_toDateStr 修正で解消） |
| `getWeeklyRanking` 派生キャッシュ | OK（type=test の集計のみ、日付は HPLog 直読み） |

## 追加調査（4/26 dryRun 後の漏れ被害者の発見）

### 事象

dryRun 第 1 回の結果、24009 岩倉寿人 と 23030 髙山遥太 が `updates` に出てこなかった。両名とも：

- Students.STREAK = 1 / 6（明らかに低い）
- Attempts シートに 4/22-4/26（または 4/21-4/26）の毎日の合格レコードあり
- 手動修正なし

→ HPLog の `type='login'` レコードが**抜けている日がある**ため、初版 recoveryがそれらの生徒の連続日数を 1 と判定してしまった。

### 仮説（ふくちさんの観察）

「ログインせずにテストを受ける」ことは論理的にあり得ない。よって Attempts に記録があるのに HPLog の `type='login'` が無い日があるなら、**ログイン処理自体が壊れていた**ことを意味する。

### コード分析による裏付け

[gas/Code.js loginStudent](../gas/Code.js)：

```js
let loginBonus = 0;
if (lastLogin !== today) {       // ← この条件で skip された場合、HPLog 'login' は書かれない
  loginBonus = 10;
  ...
  _logHP(studentId, loginBonus, loginBonus, 'login');
}
```

`_toDateStr` バグ（cache 経由で ISO 文字列を slice → 日付が JST に対して off-by-one）と組み合わさると、特定の race / cache 状態で `lastLogin === today` が誤って真になり、HPLog 書き込みがスキップされる経路が成立しうる：

- 例えば cache が JSON-roundtrip 経由で stale ISO を返す瞬間に、稀に "今日の日付" と一致してしまう状況
- もしくは loginStudent の同一日内 2 回目以降の呼び出しで cache 更新タイミングが微妙にズレ、誤って "logged in today" と判定して skip

完全な race の組み合わせの再現は困難だが、**観測された結果（Attempts はあるが HPLog 'login' が無い日が存在する）が動かぬ証拠**。HPLog のみを単一信号としていた初版 recovery は、この経路で失われた 'login' レコードを復元できない。

### 改善内容（recoverAllStudentsStreak v2）

ふくちさんの仕様確認「アプリはログインしないとテスト等の課題は出来ない」に従い、以下を **すべて "ログイン日" として扱う**：

1. HPLog の `type='login'` レコード（主信号）
2. HPLog の `type='test'/'sango'/'wabun1'` 等の他 type（apology_streak_bonus は除外）
3. Attempts シートのレコード（テスト受験＝必ずログイン後）

3 つの source の和集合からユニーク日付を取って、loginStudent と同じ streak 計算ロジックを適用する。

### 個別生徒診断のための `diagnoseStudentActivity(studentId, days)`

特定生徒の HPLog / Attempts を時系列で抽出し、各日の login/attempt/他 type の有無を可視化する関数を追加。GAS エディタで個別実行する想定：

```javascript
diagnoseStudentActivity('24009', 30)  // 24009 岩倉寿人の過去 30 日分
```

戻り値の `missingLoginDays` に「Attempts/他 type はあるが login が無い日」が列挙される → 4/24 cache バグの実害ログとして残る。

## 4:00 AM 教育日システム（4/27 から適用）

### 動機

深夜まで頑張って学習している生徒が、深夜 0:00 を跨いで再ログインしても連続が切れないように。朝 4:00 前後に起きて学習する生徒は常識的にいないため、4:00 AM を「1 日の境界」とする。

### 実装

`_todayEducationalJST()`：

- 適用開始時刻 `2026-04-27 00:00 JST` 未満：`_todayJST()` と完全に同じ挙動（4/26 中の復旧作業との互換性）
- 適用開始以降：JST 04:00 区切り
  - 0:00 ≤ JST hour < 4:00 → 前日の日付を返す
  - 4:00 ≤ JST hour < 24:00 → 当日の日付を返す

例（cutover 後）：
- 2026-04-27 03:30 JST → 教育日 `'2026-04-26'`
- 2026-04-27 04:00 JST → 教育日 `'2026-04-27'`
- 2026-04-27 23:59 JST → 教育日 `'2026-04-27'`

### 影響範囲

| 関数 | 旧 | 新 |
|---|---|---|
| `loginStudent` | `_todayJST()` | `_todayEducationalJST()` |
| `saveAttempt` | `_todayJST()` | `_todayEducationalJST()` |
| `getTodaysSet` | `_todayJST()` | `_todayEducationalJST()` |
| `_getYesterdayJST` | カレンダー昨日 | 教育日基準の昨日 |

切り替えない関数：

- `_sangoToday()` / `_sangoPrevDate()`：3 AM 区切りで既に運用中、影響なし
- `runDailyBackup`：実カレンダー日でファイル名を作る
- `_getLastWeekRange`：実カレンダー週の集計
- `recoverAllStudentsStreak`：本日（4/26）の復旧では旧基準のまま（仕様確認のため）
- `apologyStreakBonus`：固定文字列フラグキーのみ使用、日付計算なし
