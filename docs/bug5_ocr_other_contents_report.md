## 0. エグゼクティブサマリ

マイ活アプリで OCR / Vision API を使って正誤判定をしているコンテンツは **4 件**（カンジー書き / 基礎計算 / 英単語RUSH 5級書取 / 和文英訳①）。**4 件すべてが「正解情報を Vision API に渡さない自由 OCR 設計」** で、構造的にはバグ⑤と同根。

ただし**字形空間の広さ**と**生徒チェック介在の有無**で実害リスクが大きく異なる。

| コンテンツ | 文字空間 | 飛躍誤認識リスク | 生徒チェック | 実害現状 | 修正優先度 |
|---|---|---|---|---|---|
| **カンジー** | 漢字（極大） | **極高** | なし | **多数報告** | 🔴 **緊急**（Phase B） |
| 基礎計算 | 数字 + 記号（小） | 中 | なし（確認画面はあるが生 OCR は非表示） | 散発（複雑分数等） | 🟡 中期 |
| 英単語RUSH 5級書取 | アルファベット（極小） | 低 | なし（OCR 結果は失敗時のみ表示） | 5/7 空白バグ修正後は安定 | 🟢 不要 |
| 和文英訳① | アルファベット（極小） | 低 | **あり**（parsedAnswers 確認画面で撮り直し可） | 安定 | 🟢 不要 |

**結論：カンジー以外は「同根の構造ではあるが、字形空間 or 生徒チェックでリスクが抑えられている」。カンジー Phase B（案 A）を最優先で完遂し、他は当面現状維持。**

## 2. 各コンテンツの採点フロー詳細

### 2.1 基礎計算

#### 2.1.1 全体フロー

```
[生徒] 答案ノート（5問または10問）を1枚で撮影
        │
        ▼
[フロント] 1600px / JPEG 0.85 縮小 → base64 化
        │ POST: action=submitKisoAnswer, sessionId, imageBase64, hasWorkPhoto
        ▼
[GAS submitKisoAnswer]  gas/Code.js 5303 行〜
  ├─ セッション特定 → targetIds（採点対象問題 ID リスト）取得
  ├─ _kisoOcrWithGemini(imageBase64, numQuestions)   ★OCR 本体
  │   ・gemini-2.5-flash / temperature=0
  │   ・responseMimeType: 'application/json'（JSON 強制）
  │   ・出力例: {"1":"1/4", "2":"1", "3":"2√5", ...}
  ├─ studentAnswers = answersMap から番号順に配列化
  ├─ 全部空なら retake:true で返却
  ├─ 各問題を _kisoMatchAnswer(studentText, answerAllowedJson) で採点
  │   ・_kisoNormalize → 既約分数/平方根/有理化の厳格形式チェック → allowed と完全一致
  ├─ 80%以上で合格（初回提出）、再挑戦は全問正解で合格
  └─ Drive 保存（KisoPhotos）+ KisoSessions 更新 + HPLog 記録
```

#### 2.1.2 プロンプト（[gas/Code.js:3899-3918](gas/Code.js:3899)）

```
これは数学の解答用紙の写真です。番号（①②③④⑤⑥⑦⑧⑨⑩）の後に書かれた「答え」を抽出してください。

【書式ルール】
・分数は「分子/分母」の形式（例：3/4）。横線分数も縦書き分数も同じ形式に統一。
・帯分数は「整数 分子/分母」の形式、半角スペース 1 個で区切る（例：1 1/2）
・平方根は「√数字」「整数√数字」の形式（例：√3、2√5）
・指数は「^」を使う（例：2^3、x^2）
・マイナス記号は半角ハイフン「-」（例：-7、-3/4）
・「x = 5, y = -2」のような連立方程式の解は文字列のまま「x=5, y=-2」と書く

【出力ルール】
・出力は JSON オブジェクトのみ。前後の説明文や ```json などのコードブロックは絶対に含めない
・キーは番号の文字列 "1", "2", ..., "<numQuestions>"
・値は答えの文字列（読み取れない場合は空文字 ""）
・1〜<numQuestions> 番までを対象とする
・存在しない番号や読み取れない番号は空文字 "" を入れる（キー自体は省略しない）

【出力例】
{"1": "1/4", "2": "1", "3": "17/30", "4": "2√5", "5": ""}
```

#### 2.1.3 観点別評価

| 観点 | 状態 |
|---|---|
| 正解情報を Vision に渡しているか | **No**（自由 OCR、書式変換ルールのみ） |
| 画像送信単位 | **全体 1 枚**（5〜10 問混在） |
| 出力形式 | **JSON 強制**（responseMimeType + プロンプト二重） |
| 採点ロジック | `_kisoMatchAnswer`：正規化後、allowedJson の各要素と完全一致比較 |
| 表記揺れ吸収 | あり（既約分数 / √ / 全角半角等を `_kisoNormalize` で吸収） |
| 信頼度利用 | `_kisoAverageOcrConfidence` ヘルパーは存在するが、Gemini レスポンスには confidence が無いため実質未使用 |
| 生徒チェック介在 | 確認画面はあるが、表示されるのは「生徒が撮った写真」のみ。**OCR 結果（answersMap）は生徒に見せない** |

#### 2.1.4 バグ⑤と同根の構造か

**Yes（部分的に）**：プロンプトに正解情報を渡していない自由 OCR である点はカンジー・和文英訳①と同じ。

ただし**字形空間が劇的に狭い**：数字 0-9・四則記号・√・分数線・x・y・等号 などの限定セット。「字形が遠い別の文字に飛躍誤認識」は理論上発生するが、漢字に比べて頻度は遥かに低い。

#### 2.1.5 既知の対策

- 5/7 commit `4c8d960`：Gemini 2.0-flash → 2.5-flash 切替
- 5/7 commit `d13cf66`：自動リトライ機構（HTTP 429 / 5xx / 帯域系で 1 回内部リトライ）
- 5/7 commit `3d3a3f3`：写真 1000/0.6 → 1200/0.7 強化、後 1600/0.85 に追加強化
- 5/8 commit `1cad195` / `d3595ba`：数値正規化（`+2`→`2`、`2.0`→`2`、`x²`→`x^2` 等）

#### 2.1.6 実害現状

- 散発的な「分数の縦書き認識ミス」「複雑な式の番号対応ミス」報告はあったが、5/8 までの一連改修で安定。
- ふくちさんからの緊急修正報告は今のところ無い。

#### 2.2.6 実害現状

## 3. バグ⑤と同根の問題があるコンテンツ一覧

「**正解情報を Vision API に渡さない自由 OCR**」という構造的観点では、**全 4 コンテンツが同根**。

ただし「実害として表面化しているか」「飛躍誤認識の構造的リスクが高いか」で区別すると：

### 3.1 構造的リスクの定量比較

| コンテンツ | 文字空間サイズ | 単一文字種類数（概算） | 「飛躍誤認識」発生確率（推定） |
|---|---|---|---|
| カンジー | 漢字 5 級〜2 級 | 約 2,500 字 + 字形類似ペア多数 | **高**（恩恵→困田、怒鳴→焚き 等の実例） |
| 基礎計算 | 数字 + 記号 | 約 30 字種（0-9, +-×÷=/√, x, y, 等） | 中（複雑分数の縦書きで稀に発生） |
| 英単語RUSH 5 級 | アルファベット | 26 × 2 + 補助 | 低（理論上は e/c、O/0 等の混同のみ） |
| 和文英訳① | アルファベット + 日本語混在 | 同上 + ひらがな（モニョ等） | 低〜中（生徒チェックで吸収） |

### 3.2 同根構造の検出ポイント

すべてのコンテンツで以下が成立：

1. ✗ **正解情報がプロンプトに含まれない**（あるいは Cloud Vision のように指定する手段すらない）
2. ✗ **画像は答案全体を 1 枚で送信**（1 問 1 撮影 UI は無し）
3. ✗ **Vision は「自由 OCR」タスクを解いている**
4. ✗ **信頼度（confidence）情報は利用していない**
5. Vision レスポンスは「テキスト全文」or「番号別 JSON」のみ

**カンジー以外で実害が表面化していない最大の理由**：**アルファベット・数字の字形空間が漢字の約 100 分の 1** のため、飛躍誤認識の構造的発生確率が極めて低い。

#### 🔴 カンジー（緊急）

- 字形空間 + 確認画面に生 OCR 非表示 + 教育者からの実害報告（恩恵・通・怒鳴/網/齢/触 の事例）
- Phase A レポートの案 A 推奨どおり、Phase B 実装が最優先

#### 🟡 基礎計算（中期）

- 字形空間は狭いが、ゼロではない（「3」と「8」、「5」と「6」、複雑分数の認識）
- 現状は安定運用中なので緊急ではない
- 5/7-5/8 で一連の改修（モデル切替、リトライ、写真強化、数値正規化）を実施済みで、追加余地は限定的
- 案 A 風の実装をすると「allowed 配列 + 正解 canonical」を渡すことになるが、`_kisoMatchAnswer` 自体が正規化 + allowed 比較を既に行っており、OCR タスクを「allowed のどれに最も近いか判定」に切り替える価値はある
- ただし、数学では「allowed のどれにも一致しない誤答」も拾わないといけない（部分点 / 解説のため）ので、カンジーよりは設計が複雑

#### 🟢 英単語RUSH 5 級（不要）

- 構造的に飛躍誤認識リスクが極小
- 5/7 修正後、報告無し
- 案 A 適用しても効果が見込めない（既に「単語ごとの regex マッチ」で実質「ヒント付き」と等価）

#### 🟢 和文英訳①（不要）

- 生徒チェック介在の確認画面が大きな安全装置として機能
- 8 分類フィードバックで不正解時の説明も豊富
- 案 A 適用すると「Wabun1Topics の answers をプロンプトに混入」する形になるが、構造的には英語の完全一致判定にすぎず、現状で十分機能している
- もし将来「OCR が長文を絶望的に誤読する」事象が頻発したら、案 A を検討する余地はある（実装は容易、各問の `answer<N>` を埋め込むだけ）

### 4.2 横展開の可能性（カンジー Phase B 完了後）

カンジー Phase B（案 A）が成功し、Gemini Vision の「判定タスク化」が effective だと確認できた場合、**他コンテンツへの横展開**を検討できる：

- **基礎計算**：allowed の各候補と入力字形の一致判定にすれば、複雑分数の認識ミスが激減する可能性。実装難易度 M〜L（多数の表記揺れを扱う必要あり）
- **英単語RUSH 5 級**：「期待単語 N 個のそれぞれが 3 回以上書かれているか」を judgment 形式で問える。実装難易度 S。ただし**現状で問題ないため優先度低**
- **和文英訳①**：「正解 N 文と生徒の答えそれぞれの一致 / 不一致」を judgment 形式に。実装難易度 S。ただし**生徒チェックで十分**なので優先度低

→ **横展開は Phase B 完了後に「実害頻度」と「Gemini Vision 判定タスクの効果」を見て判断**するのが現実的。

## 5. ふくちさんへの確認事項

### Q1：横展開の意思決定タイミング（最重要）

カンジー Phase B（案 A）の実装後、効果が確認できた段階で：
- (a) **すぐに基礎計算にも案 A を適用したい**（横展開積極派）
- (b) **基礎計算は当面現状維持、実害報告が累積してから検討**（推奨、慎重派）
- (c) **基礎計算も含めて全 OCR コンテンツに段階的に案 A を入れたい**（中期計画として）

→ Phase B 着手前にこの方針が固まっていれば、Phase B の関数命名や共通基盤抽出を将来横展開しやすい形で書ける（例：`_kisoJudgeWithGemini` と `_kanjiJudgeWithGemini` で共通プロンプトテンプレートを使う等）。

### Q2：基礎計算の「allowed のどれにも一致しない誤答」の扱い

仮に基礎計算に案 A 適用するとして、生徒の答えが「期待 allowed のどれにも一致しない誤答」だった場合：
- (a) Vision に「allowed のどれに最も近い文字を書いたか」を返させる（学習フィードバック充実）
- (b) Vision に「all_no_match」も選ばせる（純粋に誤答として扱う、フィードバック簡素）

ふくちさん 36 年経験「不合格時こそ学習機会」哲学に合うのは (a) ですが、Vision レスポンスの解釈が複雑になります。

### Q3：和文英訳① への案 A 適用の必要性判定

現状の「生徒チェック介在」が機能している前提で：
- (a) 不要、現状維持（推奨）
- (b) parsedAnswers を Vision にも渡して、確認画面表示前に正解と一致するか先回り判定（生徒の作業量を減らす方向）

(b) は「生徒が問題を解いたか教師が分からなくなる」リスクと、「英訳学習の本質は『書く前にちゃんと考える』」教育原則の観点でデメリットが大きいと推測。

### Q4：英単語RUSH 5 級は本当に手を入れなくてよいか

確認のみ：
- ✅ 現状維持で問題なし、優先度低（推奨）
- 何か直近の苦情がある → 教えてください
