### バグ修正 6 件

#### バグ① 三語短文：同じお題が複数回連続表示
- **原因**：管理者が同じ週を2回貼り付け → SangoTopics に同 (date, level) の重複行
- **修正**：GAS 側で「(date, level) で重複排除（後勝ち）」+ フロント側で防御層
- **コミット**：`f812c01`

#### バグ② iPad ログアウト：特定操作後に効かなくなる
- **原因**：複合（overlay 居残り + iOS Safari の confirm() ブロック + 二重タップ衝突）
- **修正**：3つの防御 + Mac WebInspector 用診断ログ
- **コミット**：`55af54e`

#### バグ③ カンジー：満点なのに「惜しい！満点で合格だよ」
- **原因**：`(res && res.correctCount) || _kanjiState.yomiScore` の falsy 評価バグ
- **修正**：ロジック修正 + 文言「🎊 満点！カンジー仙人もうなずいています」
- **コミット**：`3bd87d4`

#### バグ④ 学習履歴：ログイン判定矛盾（表示問題）
- **原因**：HPLog の type='login' レコードのみで判定 → 欠落日に矛盾表示
- **修正**：推論ベース補完（学習活動があればログイン扱い）
- **コミット**：`d279dec`

#### 🔥 バグ④-本質：ログイン/HP データ欠損（サイレントエラー根絶）⭐

**v13 最大の偉業**。長年潜んでいた構造的欠陥を根治。

- **真因**：`_logHP` の try-catch がエラーをサイレントに呑む構造 + Students.HP 更新が `_logHP` の前に行われる順序
- **既知症例**：24009 岩倉さん、23030 髙山さん（過去スレで発見済、コードコメントに記録）+ v13 で清水未唄さん（3例目）
- **修正**：
  - `_logHP` 堅牢化（戻り値 + リトライ + flush + テレメトリ）
  - 全 HP 付与経路で書き込み順序逆転「HPLog 先 → Students 後」
  - 8 関数で適用：loginStudent / saveAttempt / submitSango / submitWabun1 / submitKisoAnswer / submitLison / submitKanjiKaki / submitKobunSet
- **コミット**：9 本（`0d7d702` 〜 `2f98c71`）

#### バグ⑤ カンジー OCR 誤認識（恩恵→困田、怒鳴→焚き 等）
- **真因**：Gemini Vision に正解漢字を渡さず自由 OCR させていた構造的限界
- **修正**：案A（正解候補ヒント付き判定タスク化）
  - 「自由 OCR」→「N 個の正解候補との一致判定」にタスク変更
  - JSON 形式で {no, match, studentWrote, readable} を返させる
  - readable 3 値（yes/no/blank）で「読めた／判別不能／空白」を区別
  - readable=no は再撮影誘導（採点保留）
- **KanjiSubmissions シート新設**（軽量版F'）：判定結果のテキストのみ永続化
- **コミット**：5 本（`d0c7323` 〜 `8ffeb68`）

## 🛡️ 第六章：v13 で確立した設計原則

### 案 A（正解候補ヒント付き判定タスク化）

- カンジー OCR で導入、AI に「自由 OCR」させずに「N 個の候補との一致判定」にする手法
- 構造的に飛躍誤認識を排除できる
- サンゴタン AI フィードバックも類似発想（評価軸を明示して AI に判定させる）
- **他コンテンツ（基礎計算 / RUSH5級 / 和文英訳①）への横展開は v14 以降で実害観察してから判断**

### \_logHP 堅牢化パターン

- 「**HPLog 書き込み先、Students 後**」の書き込み順序を全 HP 付与経路で徹底
- `_logHP` に戻り値を持たせ、失敗時は早期 return
- 既存データの整合性を保つ（過去欠落は救済しない、今後の発生を止める）

### 軽量版 F'（判定結果のテキスト永続化）

- 答案画像や録音は保存しない、判定結果のテキストだけシートに残す
- 容量負荷ゼロで効果測定とトラブルシュートが可能
- 適用シート：KanjiSubmissions（バグ⑤）、SangoSubmissions 拡張（サンゴタン）

### サイレントエラー禁止の原則

v13 で根絶した「try-catch + console.error だけ」のパターンを今後は許さない：

- 全ての書き込み処理は戻り値で成否を返す
- 失敗時は呼び出し元に伝える
- 重要処理（HP付与等）は失敗時にロールバックする

## 🔄 第七章：v13 で確立した運用ルール

### Claude Code 起動時のルール

1. PowerShell から起動 or タスクバー起動（プロジェクト名 / ブランチ確認）
2. 必ず **dev ブランチ**で作業
3. 終了時は `/exit` または Ctrl+C 2 回（× ボタン禁止）
4. 大規模変更前に事前確認を出させる

### Claude Code 完了報告のルール

- 各コミットの SHA を必ず提示（嘘事案防止）
- バックエンド/フロント等の大規模実装は段階的な完了報告（2 段階構成）

### 本番反映 4 ステップ

1. `git checkout main && git merge dev && git push origin main`
2. `cd gas && clasp push`
3. Apps Script エディタで F5 リロード
4. **「デプロイを管理 → 既存デプロイの鉛筆 → 新バージョン → デプロイ」**（「新しいデプロイ」ボタンは絶対 NG）

### worktree 掃除のルール

- Claude Code 起動中は worktree を消さない
- 別セッションを始める前に「`/exit` → worktree 残骸チェック → 削除 → 起動」

### スレッド管理（最重要・恒久ルール）

スレが文字数上限に近づく兆候を察知したら、作業中でも会話を遮って引き継ぎ書作成を強行提案する。
