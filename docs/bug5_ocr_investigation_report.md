## 0. エグゼクティブサマリ

カンジー書き答案の OCR 誤認識は、**「Gemini Vision に正解漢字を伝えず、答案ノート全体を 1 枚の画像として渡し、AI 任せでテキスト化させている」現状アーキテクチャの構造的限界**に起因する。プロンプト強化（5/7 補正禁止指示・5/8 厳格化・反例提示）は加点として有効だが、生徒の手書きが「字形として遠い別の漢字に飛ぶ」誤認識（怒鳴→焚き／網→称／齢→断／恩恵→困田／通→海記）を Vision 単体で根治するのは難しい。

**真因仮説の本命は「正解候補ヒント未供給」と「マルチ漢字セグメンテーション失敗」の併発**であり、対症療法（プロンプト/前処理）だけでなく、**呼び出し方そのもの（API 設計）の見直し** が必要。具体的には「正解候補を 4 択ヒントとして渡し、判定タスクに切り替える」（案 A）が最も短期で実装でき、副作用が小さく効果が大きい。

**推奨：案 A（正解候補ヒント付き判定タスク化）を最優先で採用、案 D（1 問 1 撮影 UI）を中期で検討。**

### 1.2 重要事項

・1 行に「番号 漢字」または「番号. 漢字」の形式で書かれていることが多い。

・写真には畳・机・木目・影・紙の縁・他の物体などの背景が写り込んでいる場合があります。
・紙の上に書かれた手書き文字のみが読み取り対象です。
・背景の模様や影の濃淡を文字と誤認しないでください。

・トメ・ハネ・ハライの細部の揺れは許容する。

### 2.2 プロンプトが効いていない理由（推測）

- **「字形どおり読め」と命令しても、Vision モデルは入力画像が判別困難な場合にトレーニング分布の頻出語へ引き寄せられる**（LLM の言語モデル的優先傾向）。プロンプトで抑制しても完全制圧は難しい。
- 「字形が遠い別の漢字」（恩恵→困田、通→海記）に飛ぶケースは、**プロンプト指示の射程外**で起きている。Vision がそもそも文字境界（セグメンテーション）を誤検出していると推測される。
- 「?」を返すことを許可しているが、現実には Vision モデルが「?」を選ばずに何らかの漢字を出力する傾向（命令への部分服従）。

## 3. 現状採点ロジック

### 3.1 正規化処理 `_kanjiNormalizeText(s)`（[gas/Code.js:11568](gas/Code.js:11568)）

```js
function _kanjiNormalizeText(s) {
  if (s == null) return '';
  let t = String(s);
  // 全角英数 → 半角（かな・漢字は保持）
  t = t.replace(/[Ａ-Ｚａ-ｚ０-９]/g, function(ch){
    return String.fromCharCode(ch.charCodeAt(0) - 0xFEE0);
  });
  // 空白系すべて削除（半角/全角スペース/タブ/改行/zero-width）
  t = t.replace(/[\s　​‌‍⁠﻿]+/g, '');
  return t;
}
```

- 漢字本体・ひらがな・カタカナはそのまま保持。
- 改行・空白の差は吸収できるが、**字形誤認識は吸収できない**（当然）。
- 句読点・記号は保持（不要な情報が混入し得る）。

### 3.2 照合方法：cursor 前進方式の `indexOf`（[gas/Code.js:11429-11450](gas/Code.js:11429)）

```js
const ocrNorm = _kanjiNormalizeText(ocrText);
let cursor = 0;
let correctCount = 0;
const results = expectedAnswers.map(function(e){
  const ans = String((e && e.answer) || '').trim();
  const ansNorm = _kanjiNormalizeText(ans);
  let found = false;
  if (ansNorm) {
    const pos = ocrNorm.indexOf(ansNorm, cursor);
    if (pos >= 0) {
      found = true;
      cursor = pos + ansNorm.length;
    }
  }
  if (found) correctCount += 1;
  return {
    kanjiId, no, expected: ans, isCorrect: found
  };
});
const passed = total > 0 && correctCount === total;
```

- 「OCR テキスト全体に正解の連続部分文字列が cursor 以降に存在するか」だけを見る完全部分一致。
- **OCR が誤った時点で詰む**。例：正解「恩恵」、OCR「困こん田でん」→ `indexOf('恩恵', 0)` は -1 → 不正解確定。
- 順序保護はあるが、これは「正しく OCR されている前提」での話。OCR が壊れた瞬間に意味を失う。
- **信頼度（confidence）は一切利用していない**。Gemini Vision の response からは confidence を取得していない（Vision API と違い Gemini Vision のレスポンスに symbol-level confidence は無いため、構造的に取れない）。

### 3.3 合格判定

- `passed = (total > 0 && correctCount === total)` ★全問正解必須。
- `KANJI_PASS_RATIO = 1.0`（[gas/Code.js:11046](gas/Code.js:11046)）に基づく知識系コンテンツ方針（5/8 ふくちさん哲学）。
- 不合格時の HP 付与なし。OCR ミスで 4/5 → 不合格判定 → 再撮影地獄、という運用上の苦しさが**ここで効いてくる**（合格ラインが緩い基礎計算と違い、カンジーは OCR ミス 1 つで再挑戦に飛ばされる）。

## 4. 真因仮説（複数）

### 仮説 H1：正解候補ヒント未供給（本命・最有力）

**事象**：恩恵→困田、通→海記、怒鳴→焚き 等。

**仮説**：Gemini Vision は「自由 OCR タスク」を解いており、「**この答案には正解候補 N 個のうちのどれかが書かれているはず**」という事前知識を持たない。判別困難時の最尤推定が「正解漢字」から大きく外れる。プロンプト指示は意味的制約に止まり、**確率分布の事前情報としては作用しない**。

**根拠**：
- 仮説の整合性が最も高い。Vision モデルは入力画像と訓練分布に基づく posterior を返すだけで、外部の正解集合を知らないため。
- 「字形が完全に遠い」誤認識（怒鳴→焚き 等）の説明として最も自然。Vision が「字形どおり」読もうとして失敗した結果、隣接漢字を巻き込んだセグメンテーション + 不適切な候補が選ばれた、と説明できる。
- 同じ画像でも複数の正解候補から「最も近い字形」を選べる、という制約タスクに切り替えれば誤認識は劇的に減ると予想される。

### 仮説 H2：マルチ漢字セグメンテーション失敗（併発）

**事象**：「怒鳴」（2 文字）→「焚き」（1 文字 + 仮名）の字数違い誤認識、「通」3 回→「海記」（隣接問題巻き込み？）。

**仮説**：Gemini Vision の文字境界（characters の bbox）推定が、特に手書きの「字間が詰まっている」「行ピッチが揃っていない」「線が大きく重なる」シチュエーションで誤動作する。1 文字を 2 文字に分割したり、隣接文字をマージしたりする。

**根拠**：
- 「通を 3 回書いた答案 → 海記」の事例は、3 文字あるはずが 2 文字として読まれた。セグメンテーション失敗の典型症状。
- プロンプトに「字数を変えるな」と書いても、根本的にセグメント認識が壊れていれば命令だけでは矯正できない。

### 仮説 H3：画像品質（前処理）の不足

**事象**：複雑漢字（10 画以上、例：恩恵・齢・網）の誤認識頻度が高い印象。

**仮説**：1600px / JPEG 0.85 の縮小は健全（5/8 commit `3d3a3f3` で 1200/0.7 から強化済み）だが、生徒のノート用紙の質、撮影環境（薄暗い室内、紙の影、手ブレ、ピンボケ）の影響を吸収しきれていない可能性。

**根拠**：
- 既に 1600/0.85 まで強化しており、追加余地は限定的。
- 仮にあと数 % の改善幅があっても、H1・H2 が解決しないと根治には至らない。

### 仮説 H4：プロンプト指示への部分服従

**事象**：「?」を返すべき場面で別漢字を出力する。

**仮説**：Gemini Vision は命令には従うが、「断定的な漢字を出すこと」が訓練分布の主モードであり、「?」を返す probability が低い。プロンプト命令が softmax の温度的に作用するのみ。

**根拠**：
- 5/7 厳格化（補正禁止）と 5/8 反例提示でも残っている事象であることから、プロンプト改善の単独効果には限界がある。
- 「temperature: 0」は採用済みだが、これは出力決定性の話で、サンプリングの偏りそのものは変えない。

### 仮説 H5：採点ロジックの脆弱性（補助的）

**事象**：cursor 前進方式の `indexOf` が、ノイズ（誤読された他文字）に挟まれた場合に検索に失敗する可能性。

**仮説**：例えば正解「恩」「恵」を期待して、OCR テキストが「1.困こん 田でん 2.（次の答え）...」のとき、`indexOf('恩', 0)` は不一致、`indexOf('恵', cursor)` も不一致で、両方 ❌。これは事実上 OCR ミスを採点で覆せない設計を意味する。

**根拠**：
- 採点側でできることは限定的（漢字の異体字許容、許容漢字テーブル等）。本質的には OCR 品質の問題。
- 補助的な改善余地として、「番号行ごとに分割し、その問題の正解候補と『最近傍の漢字』を比較する」近傍探索方式は検討余地あり（案 C）。

## 5. 改善方針案（A/B/C/D...優先順）

### 案 A：正解候補ヒント付き判定タスク化（★ 最優先推奨）

**概要**：Gemini Vision のタスクを「自由 OCR」から「**N 択判定**」に切り替える。プロンプトに各問題の正解候補を埋め込み、「番号 X の答え欄に書かれた漢字は、以下のうちどれに最も近いか」を問う。出力形式を JSON にして parse する。

**具体例（プロンプト例）**：

```
これは生徒の手書き答案の画像です。以下の各問題について、答え欄に書かれた字形が
「正解の漢字」と一致するか判定してください。

問題1: 期待する漢字 = "恩恵"
問題2: 期待する漢字 = "通通通"  ← 3回連続
問題3: 期待する漢字 = "怒鳴"
問題4: 期待する漢字 = "網"
問題5: 期待する漢字 = "齢"

各問題について、以下の JSON 配列で出力してください（前置き・説明なし）：
[
  {"no":1, "match":true,  "studentWrote":"恩恵"},
  {"no":2, "match":false, "studentWrote":"?"},
  ...
]

判定ルール：
・字形の偏旁・冠脚が完全に一致するなら match=true
・部首が違う、字形が遠い、判読困難な場合は match=false で studentWrote には「?」または読み取れた近似字を入れる
・字形がほぼ一致するがトメ/ハネ/ハライだけ違う場合は match=true（部首ベース判定）
・正解候補に「寄せる」ことは禁止。書かれた字形が違うなら必ず false。
```

**期待効果**：
- 「自由 OCR」より遥かに易しいタスク（候補比較）。
- 正解漢字が事前情報として与えられることで、Vision モデルは「期待される字形と入力画像の類似度」だけを評価すればよく、確率分布が劇的に絞れる。
- 「怒鳴→焚き」「網→称」のような飛躍誤認識は、判定タスクでは「焚きは怒鳴に似ているか？ → No」となるため激減する見込み。
- 副次効果：OCR テキスト全文を取得しなくても採点できる（判定結果のみ取得）。

**実装難易度**：M
- `_kanjiOcrWithGemini` を `_kanjiJudgeWithGemini(imageBase64, expectedAnswers)` に置き換え（独立関数として共存も可）。
- プロンプトを動的構築：expectedAnswers 配列から問題リストを生成して埋め込み。
- response の JSON parse + バリデーション + フォールバック。
- 採点ロジックは cursor 前進方式から「`results[i].match` を見るだけ」にシンプル化。
- 既存 `submitKanjiKaki` のレスポンス形（results / passed / ocrText）と互換性を保つラッパー実装可能。

**副作用リスク**：
- 「答え欄に何も書いていない」を Vision がどう判定するか要検証（true を返してしまうリスク）。プロンプトに「答え欄が空白なら match=false」を明記する必要あり。
- 「studentWrote」フィールドに何が入るか Vision 任せ。不合格時の答え合わせ表示で「あなたが書いた内容」を出すには、ここを信頼する必要がある。
- response が JSON にならないケース（コードブロック・前置き付き）への耐性（既存処理を流用）。
- レート・トークン使用量は短いプロンプト + 短い JSON 出力なので、現状とほぼ同等以下と推定。

**推奨/非推奨**：★ **強く推奨**。最も短期で実装でき、効果が大きく、副作用も限定的。本命対策。

---

### 案 B：プロンプト追加強化（補助）

**概要**：現状プロンプトに「判別困難時は必ず空文字でなく `?` を返す」「答案領域の中央 80% のみ読む」「字形比較は『部首 4 構成要素（偏旁冠脚）』ベースで」等の細則を追加。

**期待効果**：
- 漸進的な精度向上。1〜数 % のジャンプが期待できる。
- ただし**飛躍誤認識（怒鳴→焚き）の根治には届かない**見込み（既に 5/7・5/8 で強化済みなのに残っている事実）。

**実装難易度**：S
- プロンプトを書き換えるだけ。リスクは最小。

**副作用リスク**：
- プロンプトが長くなりすぎると Vision の注意散漫リスク（指示の優先順位が曖昧化）。

**推奨/非推奨**：単独採用は **非推奨**（限界が見えている）。案 A の補助として、案 A 内のプロンプトの中で必要な部分だけ吸収する形で活用するのが現実的。

---

### 案 C：OCR 後の近傍探索採点（採点ロジック強化）

**概要**：cursor 前進方式の `indexOf` を、「OCR テキストを行（番号）ごとに分割し、その番号の答案テキストと正解漢字を編集距離 + 漢字異体字テーブルで近傍判定する」方式に切り替え。

**期待効果**：
- OCR が小さく誤った場合（例：「己」→「已」、「未」→「末」）の救済。
- ただし**飛躍誤認識（怒鳴→焚き）は近傍とは言えず救えない**。

**実装難易度**：M〜L
- 編集距離計算（Levenshtein 等）を漢字単位で実装。
- 漢字異体字テーブル（旧字体／新字体／よくある誤字ペア）を整備。
- 番号行へのテキスト分割（プロンプトに「1.」形式を強制している前提）。

**副作用リスク**：
- 「異体字許容で全問正解にする」と教育的価値が薄れる懸念（書道としての厳密性が落ちる）。
- 教育設計原則（5/8 知識系コンテンツ = 全問正解）と整合性は要議論。

**推奨/非推奨**：**保留**。案 A が機能すれば不要となる可能性が高い。案 A 採用後の継続観察結果次第で検討。

---

### 案 D：1 問 1 撮影 UI（中期改修）

**概要**：「ノート全体を 1 枚で撮影」から「1 問ずつ撮影 → 答案を 1 文字（送り仮名込み）で切り出す」UI に変更。

**期待効果**：
- マルチ漢字セグメンテーション失敗（仮説 H2）を構造的に排除できる。
- 1 問あたり 1 漢字を Vision に渡せば、判別精度が劇的に上がる（Vision は単漢字判別は得意）。
- 案 A と組み合わせれば、「単漢字画像 vs 正解候補 1 つ」の最も簡単なタスクになる。

**実装難易度**：L
- 撮影 UI を 1 問ずつ撮るフローに改修（フロント大規模変更）。
- 撮影回数が問題数分に増えるため、生徒の体験が変わる（10 問なら 10 回撮影）。
- 既存「ノートを撮って送る」体験との切り替えタイミング要検討。

**副作用リスク**：
- 撮影回数増加に伴う体験悪化。
- 生徒のスマホ／タブレット運用負荷増（特に塾の共用 iPad 環境）。
- 既存生徒の慣れたフローを壊す。

**推奨/非推奨**：**中期検討**。短期は案 A で対応し、案 A でも残るケース（特に複雑漢字／字間が詰まった答案）の頻度を計測してから判断。

---

### 案 E：採点結果に「OCR 信頼度バナー」表示（補助 UX）

**概要**：採点結果画面の「答え合わせ」セクションに、「これは AI の読み取り結果なので、書いた字とずれている場合があります」というヒント文（既存）を強化し、**さらに「読み取りに自信がありませんでした」マークを問題単位で表示**する。

**期待効果**：
- 教育的フォロー：生徒が「自分は正しく書いたのに採点されない」と感じたとき、AI の限界を理解できる。
- 答案そのものは保存していないので、事後検証はできないが、「次回はもっと大きく書こう」等の行動変容を促せる。

**実装難易度**：S（案 A 採用後）
- 案 A の judgment レスポンスに `confidence` フィールドを Vision に出力させ、それを UI に反映するだけ。
- 案 A なしでは Gemini Vision のレスポンスから confidence は構造的に取れない（Vision API と違う）。

**副作用リスク**：
- 「信頼度低 = 採点甘く」を期待する保護者問い合わせのリスク。「信頼度低でも採点結果は正」というポリシーを明示すべき。

**推奨/非推奨**：案 A の追加オプションとして **検討**。

---

### 案 F：KanjiSubmissions シート新設で答案永続化（運用基盤）

**概要**：書き答案の写真と OCR テキストを Drive + シートに永続化。事後検証・パラメータ調整・誤判定報告対応の基盤とする。

**期待効果**：
- 生徒から「正しく書いたのに不合格にされた」報告があったとき、画像 + OCR テキスト + 採点結果を見て真因確認できる。
- AI モデル切替時の effectiveness 比較ができる。
- 教師の確認画面（admin.html）で抜き打ちチェック可。

**実装難易度**：M
- KisoSessions / 答案写真の保存パターン（既存基盤、CLAUDE.md #103）を流用可能。
- 15 日自動削除（リスオン録音と同パターン）も流用。

**副作用リスク**：
- 個人情報・容量（多数の生徒 × 多数のセッション × 写真）。
- リスオン録音の管理画面パターンと同じ運用負荷。

**推奨/非推奨**：**中期検討**。本バグの根治とは独立の運用基盤強化案として位置づけ。

## 6. ふくちさんへの確認事項（実装着手前に決めるべきこと）

### Q1：採用案の確定（最重要）

★ **案 A（正解候補ヒント付き判定タスク化）を最優先で採用するか？**
- Yes → Phase B で _kanjiJudgeWithGemini 実装に進む。
- No / 別案 → 案 B〜F のどれを採るか。

### Q2：案 A 採用時の互換性ポリシー

`submitKanjiKaki` の戻り値 `ocrText` は、生徒の不合格画面「あなたが書いた内容」表示に使われている（[index.html:6601](index.html:6601)）。案 A では Vision が JSON を返すので、フロントに何を表示させたいか確認：

- (a) 各問の `studentWrote` を「あなたが書いた内容」として 1〜N 行で表示する（推奨）。
- (b) 「あなたが書いた内容」の表示自体を廃止し、`results[].isCorrect` のみで答え合わせ。
- (c) 旧来通り全文 OCR テキストも欲しい → 案 A + 補助で OCR 実行（API コスト 2 倍）。

### Q3：「matchの基準」のさじ加減

案 A プロンプトの「match=true / false」判定基準について、教育者視点で：

- (a) **厳格モード**：偏旁完全一致のみ true、トメ・ハネ・ハライの揺れは true、字形差があれば false（教科書準拠の書き取り採点）。
- (b) **緩和モード**：書道の繊細な差は許容、別漢字に飛んでなければ true（OCR ミス耐性重視）。
- (c) **アダプティブ**：5 級は (b)、3 級以上は (a) に切替。

ふくちさん 5/8 哲学「丁寧に正しく書く」を反映するなら (a) が自然。

### Q4：「答え欄が空白」の挙動

生徒が答案を書かなかった問題について：
- (a) 必ず false 判定（推奨、教育的）。
- (b) Vision の判断に任せる（リスクあり）。
- (c) 採点前にフロントで「空白問題あり」と通知し、撮り直しを促す。

### Q5：案 D（1 問 1 撮影）の優先順位

案 A 実装後、残存誤認識の頻度を見て案 D を検討する想定でよいか？
- Yes → 短期は案 A、中期に効果測定して案 D 判断。
- No、案 D 並行で進めたい → 案 A + D の二段階リリースを並行設計。

### Q6：答案画像／OCR テキストの永続化（案 F）

- (a) 今は無くてよい（採点結果が画面に出る瞬間しか確認できなくてもよい）。
- (b) 案 F を別タスクで進めたい（運用基盤として欲しい）。
- (c) 案 A 採用時、API レスポンス（match 結果）だけは KanjiSubmissions に残す軽量版 F'。

### Q7：実装着手のスケジュール感

- (a) 即着手（次セッションで Phase B 実装）。
- (b) 仕様レビュー後（このレポートを 1〜2 日吟味してから）。
- (c) 他の優先タスク（講師ログイン Phase 2、生徒メッセージ機能等）の合間に進める。
